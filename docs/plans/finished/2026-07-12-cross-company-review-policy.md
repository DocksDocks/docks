# Make cross-company plan review the strong default

## Goal

Promote independent plan review from the current *optional, single, picker-gated* cross-tool second opinion to a **strong, availability-aware default**: every plan drafted with plan-manager is red-teamed, before execution, by **(a) the best available model of the OTHER company** and **(b) a second independent reviewer from the author's company, selected by that company's resolved tier list** — both read-only, findings-only, reconciled by the orchestrator. It is a **strong default, not a hard block** (waivable per-plan; degrades gracefully by availability so a single-subscription user is never blocked), and schema v1 runs on the portable CLI baseline (`codex exec -s read-only` / `claude -p --permission-mode plan`). Session-relay is not a canonical review transport until its binary has a dedicated non-writing reviewer mode; docks plan-lifecycle does not depend on it. Success = the plans contract + its plan-init template + the three plan skills encode author identity, policy precedence, ordered tiers, deterministic degradation/waiver outcomes, canonical receipts, pre-execution gating, sole-writer ownership, and two-reviewer attribution — self-demonstrated by this plan carrying a current-input X bootstrap record plus a successful or explicitly degraded S result.

## Context & rationale

Verbatim user decisions (2026-07-11, via picker + prompt):
- **Enforcement = strong default, availability-aware.** "strong default, but of course when available right? some people don't have double subscription." → NOT a hard block; runs when the model is available; a user with only one company's subscription runs the same-company leg as their independent review, never blocked.
- **Scope = every plan (default posture).** "Every plan, no exceptions." → the review applies to every plan by default; the strong-default waiver + availability degradation (not a stub carve-out) provide proportionality, so a genuinely trivial stub or a single-subscription run naturally executes fewer/zero paid legs.
- **Doctrine home = in the skill, runtime-global-overridable.** "in the skill itself and by global settings it can be overwritten maybe?" → ship model-AGNOSTIC orchestrator doctrine in the skill; already-loaded runtime-global guidance can override the dated default through `ReviewPolicyInput`, without a new consumer config/env contract in this repo.
- **Cross-company consent = standing authorized for this user.** "i always approve cross-review company, dont need to ask that" → a runtime/global `always` authorization suppresses Docks' own consent picker for cross-company review. This standing consent does **not** bypass a host/platform security denial; a denied launch is recorded as a degraded leg with the exact reason, never retried through an alternate export path.
- **Author intent (prompt):** gpt-5.6-sol is the strongest reasoner and is a sensible interactive orchestrator; an independent same-company leg may be an in-session subagent, while a cross-company leg uses the portable fresh CLI. A plan authored by one company is reviewed by the other company's best plus a same-company second instance. Relay remains useful for ordinary red-team sessions, but its current branch-guard prefix makes it unsuitable for the sealed non-git bundle in this policy.

Why: the mechanical pieces already exist (one-shot reviewer legs, the red-team pair); this makes independent review the default while staying portable and cost-aware. Cross-company diversity catches blind spots a model family shares; the same-company second instance is the independent leg when only one company is available and a tie-breaker otherwise.

## Environment & how-to-run

- Node 24 + pnpm (`corepack enable && pnpm install --frozen-lockfile`). Validators: `node scripts/ci.mjs`. After any SKILL.md edit, refresh `metadata.updated` + `content_hash` (ci.mjs enforces the hash) and keep bodies ≤500 lines.
- **Portable review baseline (no plugin dependency):** after sealing the bundle and computing `bundle_sha256`, plan-manager JCS-encodes the complete request. The runner uses an argv API (never shell interpolation) to append a literal readable block `REQUEST_JCS_BEGIN\n<compact-json>\nREQUEST_JCS_END` to both prompts. Reviewers copy that JSON object into `ReviewerOutput.request`; the collector JCS-canonicalizes the echoed object and byte-compares it with the source. No decoding or byte-perfect LLM transcription is required, and the request remains outside the bundle to avoid self-reference. Codex argv includes `exec -C <bundle> --skip-git-repo-check -s read-only ... --output-schema <schema> -- <prompt>`; Claude argv runs with cwd=bundle and includes `-p --permission-mode plan ... --json-schema <schema-json> --output-format json -- <prompt>`. The bundle is non-git and Claude cannot run `git show`. Never resume; every leg is fresh. GNU hosts may use `timeout 600`; elsewhere use a monotonic tool deadline.
- **Availability probes:** Codex: `command -v codex` then `codex login status`; Claude: `command -v claude` then `claude auth status`. Model availability is attempt-as-probe in the ordered tier list; `unknown model`/entitlement failures fall through once to the next listed model without counting as the leg's retry.
- **Relay boundary for schema v1:** canonical policy review rejects `orchestrator_preference=relay`. The current relay spawn prefix requires a separate Git branch, while this bundle is deliberately non-git and read-only; skill prose cannot override that binary guardrail. Step 5 updates the relay skill to say so. A future binary-level reviewer mode, binary rebuild/version/release, and end-to-end tests are a separate plan and release approval.
- **Immutable request bundle:** after committing the candidate, plan-manager fixes `reviewed_commit`, renders the canonical plan as `plan.review.md`, and exports affected paths into `/tmp/docks-plan-review/<request_id>/`; the raw plan blob is not included, so reviewers cannot accidentally read bytes outside `input_sha256`. `manifest.json` maps the logical plan path to `plan.review.md`, includes `reviewed_commit`, `input_sha256`, sorted requested paths with `state:file|directory|absent`, and sorted flattened present/generated-file `{path,mode,sha256}` entries including the plan view and reviewer schema. CREATE/deleted paths are absent tombstones; the logical plan must exist. The manifest contains no bundle hash. The helper hashes canonical manifest bytes plus each present/generated file's length-prefixed bytes; only request/result/receipt carry `bundle_sha256`. Symlinks are mode `120000`+target bytes and never followed; escapes, submodules, duplicates, commit/tree mismatches, or post-seal mutation STOP. The helper chmods the bundle read-only. Accepted findings destroy the old bundle, then commit/repeat.
- **Product policy vs host policy:** resolve the `ReviewPolicyInput` below from the already-loaded user/runtime-global instruction context; skills do not read a new env var or consumer config file. Host sandbox/export policy remains an outer gate. An explicit host denial is `platform_denied`; it is never retried through another transport. An ambiguous launch failure is `unavailable_unknown`, never guessed to be policy denial.

## Interfaces & data shapes

- **Author identity (new base plan frontmatter, captured at draft creation):** `review_author_company: openai | anthropic | unknown`, `review_author_tool: <string>`, `review_author_model: <string>`, `review_author_effort: <string>`. A legacy/unknown company triggers one picker before first review and is then persisted; X = other company, S = an independent reviewer selected from the persisted company's resolved tier list (not necessarily the exact author model). This plan records the originating Claude session as `anthropic/claude/unknown/unknown`; gpt-5.6-sol is the later orchestrator, not the author.
- **Resolved policy input (logical input supplied by the orchestrator after normal instruction precedence):** `ResolvedReviewPolicy = { schema: 1, cross_company_consent: always | ask | never, zero_reviewer_policy: ask | proceed | block, orchestrator_preference: auto | in_session | cli, openai_tiers: [{ model, effort, transports: [in_session | cli] }], anthropic_tiers: [{ model, effort, transports: [in_session | cli] }], provenance: { cross_company_consent: source, zero_reviewer_policy: source, orchestrator_preference: source, openai_tiers: source, anthropic_tiers: source } }`, where each `source = current_user | runtime_global | skill_default`; no extra provenance key is legal. Defaults are expanded before JCS/hash; tier and transport order are semantic. Selection happens once before attempts; an authoritative denial never switches transport. Results/receipts echo the exact snapshot+hash and reuse re-resolves it. `always` suppresses only X export consent. Current-turn user > runtime-global > skill default.
- **Ordered model-tier resolver (dated 2026-07; skill text says "check the current tier list"):**
  - OpenAI best: `gpt-5.6-sol` effort `xhigh`, eligible for `[in_session,cli]`.
  - Claude best: `fable` effort `high`, then `opus` effort `max`, both eligible for `[in_session,cli]`.
- **Leg + attempt result enums and ordered classifier:** `LegResult = passed | waived | not_authorized | unavailable_auth | unavailable_model | timed_out | platform_denied | failed_unparseable | unavailable_unknown`; `AttemptResult = passed | auth_failed | model_unavailable | deadline_exceeded | platform_denied | transient_transport | nonzero_exit | signaled | unparseable`. Classify in order: exact matching waiver → `waived`; product consent denies X → `not_authorized`; authoritative policy denial → `platform_denied`; auth preflight fails → `unavailable_auth`; every transport-eligible tier attempt is `model_unavailable` → `unavailable_model`; monotonic deadline → `timed_out`; exit 0 with schema-invalid output → `failed_unparseable`; success + schema-valid output → `passed`; every other terminal sequence → `unavailable_unknown`. One transient retry exists **per leg, not per tier**, and only repeats the same model/transport after pre-output typed `EAGAIN`, `ETIMEDOUT`, or `ECONNRESET`. Thus attempts are ordered and bounded `0..eligible_tier_count+1`; the extra slot is used at most once. Model-unavailable fallthrough consumes an attempt but not the retry. Strings, output-started failures, deadline expiry, and other results never retry.
- **Two-reviewer attribution grammar (extends `docs/plans/AGENTS.md`), leg-namespaced IDs, reasons preserved:**
  ```markdown
  Cross-check (<date>): [X: <company-B> <model> <effort>] N findings (sev) — accepted X<ids> / rejected X<ids> (one-line reason each); [S: <company-A second> <model> <effort>] M findings — accepted S<ids> / rejected S<ids> (reason each); [<author>] independently verified <X/S ids> against source before accepting.
  DISAGREEMENT: <topic> — [X<id>] <pos> / [S<id>] <pos>. Kept: <choice> — decided by <orchestrator|user via picker>, because <one line>.
  ```
- **Availability-degradation matrix (strong default, never a subscription hard block):** both passed → dual review; one passed → record the other exact outcome and proceed; zero passed → apply `zero_reviewer_policy`. Draft phase preserves the plan's pre-review non-executing state: ordinary new plans remain `planned`, triggered new plans remain `scheduled`; `ask` surfaces a decision without firing, `proceed` writes a degraded eligible receipt, and `block` writes an ineligible receipt without rewriting scheduled to planned. Completion remains `in_review`; `ask` waits, `proceed` writes degraded evidence, `block` keeps `review_status:null`. No completion outcome rolls back. `cross_company_consent=never` records X not-authorized and never suppresses S.
- **Prompted decision evidence (closed union):** `ReviewDecisionEvidence = XConsentDecision | ZeroReviewerDecision`; `XConsentDecision = { schema:1, kind:x_consent, decision:allow | deny, actor:string, reason:string, at:ISO, request_id:uuid, input_sha256:64hex }`; `ZeroReviewerDecision = { schema:1, kind:zero_reviewer, decision:proceed | block, actor:string, reason:string, at:ISO, request_id:uuid, input_sha256:64hex }`. All strings are non-empty and objects reject extra keys. `cross_company_consent=ask` requires X evidence before attempt; zero-passed+ask requires zero-review evidence before outcome. Standing/configured choices use policy+provenance and require null decision evidence.
- **Per-plan waivers (new frontmatter):** `ReviewWaiver = { phase: draft | completion, input_sha256: <64hex>, legs: [X | S], actor: <string>, reason: <non-empty string>, at: <ISO datetime> }`; `review_waivers: [ReviewWaiver]`. plan-manager writes one only from an explicit current-user decision; consent `never` is not a waiver. The shipped helper normalizes duplicate-free legs to `[X,S]`, JCS-encodes, and hashes the waiver. Uniqueness key `(phase,input_sha256,leg)` permits exactly one match; duplicate/conflicting entries STOP without a receipt. Every waived leg embeds the normalized snapshot/hash and reuse requires the exact persisted unique waiver; removal, duplication, reassignment, or field mutation invalidates it.
- **Immutable request envelope:** `ReviewRequestEnvelope = { schema: 1, request_id: <uuid>, phase: draft | completion, lifecycle_intent: none | start | schedule_fire | auto_execute, reviewed_commit_or_head: <40hex>, planned_at_commit:null|<40hex>, execution_base_commit:null|<40hex>, diff_sha256:null|<64hex>, acceptance_inventory_sha256:null|<64hex>, input_sha256: <64hex>, bundle_sha256: <64hex>, author:{company:openai|anthropic,tool:string,model:string,effort:string}, policy: ResolvedReviewPolicy, policy_sha256: <64hex> }`. Draft completion fields are all null. Completion fields bind the exact planned/start/reviewed range, canonical binary diff, and source-derived acceptance inventory. One envelope is created only after the plan transition/draft input is committed and the bundle is sealed; both legs, every result, and every persisted leg echo it byte-exactly. Author identity is resolved before request creation, so X/S tier validation never depends on ambient executor identity. A mismatched/stale envelope or bundle is a schema failure, not a review outcome.
- **Bootstrap evidence before the helper exists (all objects recursively closed):** `BootstrapReviewRecord = { schema:1, kind:bootstrap_not_reusable, plan_path:string, reviewed_commit:40hex, plan_blob_sha256:64hex, X:BootstrapX, S:BootstrapS }`; `BootstrapX = { tool:string, model:string, effort:string, verdict:ready|not_ready, score:int_0_100, findings_sha256:64hex, reviewed_at:ISO }`, where the hash is SHA-256 of UTF-8 JCS findings. This plan needs only two exact `BootstrapS` variants: `BootstrapSPassed = {result:passed,selected:{model:string,effort:string},verdict:ready|not_ready,score:int_0_100,findings_sha256:64hex,reviewed_at:ISO}` or `BootstrapSPlatformDenied = {result:platform_denied,attempted:false,selected:null,denial_source:sandbox|managed_policy|runtime_policy,reason:string,reviewed_at:ISO}`; strings are non-empty and no other key/result is legal. One compact record is added only after a final candidate gets a no-input-change X verdict; it binds that candidate commit/blob. The record-only commit must strip to the identical canonical view. It is historical proof only, never a v1 eligibility receipt. A5 validates it separately from post-implementation conformance.
- **Attempt + raw leg evidence (closed objects; unknown keys reject):** `AttemptEvidence = { schema: 1, model: string, effort: string, transport: in_session | cli, started: bool, output_started: bool, result: AttemptResult, exit_code: i32 | null, signal: string | null, child_id: string | null, denial_source: sandbox | managed_policy | runtime_policy | null, retry_cause: transport_EAGAIN | transport_ETIMEDOUT | transport_ECONNRESET | null, timeout_mode: gnu_timeout | orchestrator_tool | null, timeout_seconds: 600, reason: string, stdout_sha256: 64hex | null, stderr_sha256: 64hex | null }`. `RawLegEvidence = { schema: 1, leg: X | S, request: ReviewRequestEnvelope, result: LegResult, attempts: [AttemptEvidence], selected: {model:string,effort:string,transport:in_session|cli} | null, reviewer_output:{verdict:ready|not_ready,score:u8,confirmations:[string],structured_output_sha256:64hex}|null, findings: [ReviewerFinding], findings_sha256: 64hex | null, severity_totals: {high:u32,medium:u32,low:u32}, waiver: ReviewWaiver | null, waiver_sha256:64hex|null, decision_evidence: ReviewDecisionEvidence | null, reason: string | null }`. Passed requires selected equal to the final passed attempt, an exact reconstructed+hashed reviewer output, and `findings_sha256=sha256(JCS(sorted findings))` even for `[]`; `not_ready` is ineligible with no schema-v1 override. Attempts use one transport, exact per-leg company tier order, at most one immediate same-tuple transient retry, and no post-terminal attempt. Non-passed legs carry no reviewer output.
- **Reconciliation evidence:** the shipped helper sorts findings by unique leg-prefixed ID (`X1…`/`S1…`), JCS-encodes the list, and defines `findings_sha256`. `ReconciliationEvidence = { accepted: [<id>], rejected: [{ id, reason }] }` is produced only by plan-manager after independent validation; sets are disjoint/duplicate-free, every ID exists, every reproduced finding appears exactly once, and rejection reasons are non-empty. A persisted leg is `{ request, raw, reconciliation }`; plan-review never constructs reconciliation.
- **Versioned reviewer output and run results:** `ReviewerFinding = { id: X<u32> | S<u32>, severity: high | medium | low, section: string, path: string | null, locator: string | null, defect: string, fix: string, evidence: string }`; `ReviewerOutput = { schema: 1, leg: X | S, request: ReviewRequestEnvelope, verdict: ready | not_ready, score: u8_0_100, findings: [ReviewerFinding], confirmations: [string] }`. Every nested object has `additionalProperties:false`; the helper seals distinct schemas and hashes the full structured object into raw evidence. `AcceptanceInventory = {schema:1,criteria:[{id:A<u32>,command:string,expected:string}]}` is nonempty, ordered, derived from the canonical plan, and hash-bound. `PrimaryCompletionEvidence.acceptance` must cover it exactly once, in order, with identical IDs/commands/expected values. `CompletionVerdict = regressed` when either passed reviewer returns `not_ready`, CI fails, regressions is non-empty, or a high primary finding exists; otherwise `passed` requires `goal_met=yes` and every acceptance `met=true`; all other cases are `partial`. `DraftRunResult` retains `pre_execution_eligible`; draft receipts persist it. `CompletionRunResult` and completion receipts persist the inventory/hash and derived `completion_verdict`, and apply/ship require frontmatter `review_status` to equal it. All objects are closed; missing, extra, mistyped, cross-leg, request-mismatched, verdict-discarding, vacuous/invented acceptance, or status-mismatched evidence rejects.
- **Dependency-free shipped canonicalizer/validator:** `plugins/docks/skills/productivity/plan-review/scripts/review-policy.mjs` is the sole consumer implementation, runs on Node 20+, and imports only `node:` built-ins—never the author repo's `yaml` devDependency or consumer `node_modules`. It parses the deliberately closed plans-frontmatter grammar: top-level string keys; quoted/plain/null/boolean/safe-integer scalars; flow scalar arrays used by tags; indented scalar block arrays used by paths; and `review_waivers` as one-line strict JSON/JCS array. Nested YAML maps, arbitrary YAML objects, floats, aliases, tags, duplicate keys, non-string keys, tabs, lone surrogates, and out-of-range integers reject. The plan contract/template/manager must emit only this grammar. Canonical JSON follows RFC 8785 JCS; strings are not normalized and property order is UTF-16 code-unit order. The helper owns the plan-view renderer, bundle manifest/hash, policy/waiver/finding/request/result validation, reviewer JSON Schema, receipt validation, and SHA-256. Author tests execute that exact helper; a temp consumer containing only the plugin payload and no root package manifest/node_modules must pass.
- **Canonical draft view + receipt:** the helper rejects BOM/invalid UTF-8/CR-only input and normalizes CRLF→LF. It removes exactly these frontmatter keys: `updated`, `status`, `started_at`, `in_review_since`, `blocked_reason`, `blocked_since`, `assignee`, `review_status`, `ship_commit`, `execution_base_commit`, and `review_waivers`; execution-base/waiver hashes bind separately. No other key is excluded. It retains all ordinary Self-review/Review prose and removes only exact outside-fence compact-JCS one-line machine records beginning `Bootstrap-review-record: `, `Review-receipt: `, or `Completion-review-receipt: `; at most one record of each kind is legal, while machine-looking text inside Markdown fences remains ordinary hashed prose. It emits `plan.review.md` as `frontmatter-jcs + "\n" + retained-body-with-one-terminal-LF`; reviewers consume and hash exactly it. Store compact JCS inside Self-review: `Review-receipt: {"schema":1,"phase":"draft","request":<ReviewRequestEnvelope>,"input_sha256":"<64hex>","reviewed_commit":"<40hex>","author":<request.author>,"policy":<ResolvedReviewPolicy>,"policy_sha256":"<64hex>","X":<persisted-leg>,"S":<persisted-leg>,"reproduced":[FindingEvidence],"decision_evidence":<ReviewDecisionEvidence|null>,"outcome":"dual|single|zero_degraded|blocked","reviewed_at":"<ISO>"}`. Any ordinary prose edit invalidates reuse; only enumerated lifecycle/waiver fields and exact machine records do not.
- **Public entry, pre-execution gate, and sole-writer protocol:** `new plan`, `review plan`, `start`, scheduled fire, and `auto_execute` all route through main-context plan-manager. Creation first commits the current contract's non-executing state (`planned`, or `scheduled` when a trigger exists), then calls `prepare(none)`; review never changes that state. `prepare(intent)` seals input/bundle and returns `NeedsMainReviewDispatch`; main dispatches once; `apply` re-resolves/re-hashes, validates/reconciles/writes the receipt, then applies `start|schedule_fire|auto_execute` exactly once only when eligible. Missing/stale/blocked receipts STOP before `ongoing`. Scheduled block/ask remains scheduled but cannot fire; ordinary block/ask remains planned. Noninteractive ask surfaces pending decision. Manager wrappers only prepare/apply and never dispatch. plan-review is internal, evidence-only, and no other workflow may launch the same phase/hash.
- **Canonical completion input + disposable verification:** start first commits only the plan transition and captures its exact SHA as `execution_base_commit`, then records it in a second plan-only commit before work. Completion order is: mark steps done → commit plan-only `in_review` transition → validate `planned_at_commit → execution_base_commit → reviewed_head` ancestry and exact plan-only first-start transition → assert scoped clean → snapshot complete tracked/untracked/ignored content plus git-dir/common-dir metadata → seal/re-verify canonical input, `execution_base_commit..reviewed_head` binary diff bytes, and ordered acceptance inventory → run X/S as read-only findings-only legs with bundle verification before launch and after each leg → in writable main context create fixed `/tmp/docks-plan-verify/<request_id>` via unlinked clone and run exact acceptance/CI/reproduction → cleanup accepts only the closed prepare identity, rechecks owner/mode/path/token/sentinel/head/tree/snapshot, deletes the exact request child, and revalidates original immutability. Raw plan requested paths and caller-selected roots are invalid. Completion receipts include all identities/hashes and the derived verdict; ship-time reuse requires input/execution-base/diff/inventory/original/policy plus matching frontmatter `review_status` unchanged.

## Known gotchas

- Relay `spawn` returns at birth-registration, BEFORE findings exist — never advance the lifecycle on a bare spawn; require `--watch`+bus-reply or use the synchronous CLI leg.
- Claude `--permission-mode plan` cannot execute `git show`; immutable review therefore depends on the sealed bundle, not a commit named only in prose.
- Current relay spawn always injects branch/write guardrails, so it cannot host the sealed non-git schema-v1 bundle; use the portable CLI and reject relay preference.
- Two reviewer lists must namespace finding IDs (X#/S#) and preserve per-finding accept/reject reasons, or reconciliation is ambiguous.
- A receipt stored inside the plan can invalidate a naïve whole-file hash. Use the canonical exclusion algorithm exactly; any substantive section change still invalidates it.
- Product consent and host sandbox policy are different signals. `always` is not a sandbox override, and host denial is not permission to try a second export transport.
- A standing consent or zero-review decision is part of the resolved policy hash. A receipt minted under yesterday's policy cannot authorize today's run after any current-user/runtime-global/default tier or provenance change.
- Plugin-shipped Claude agents cannot enforce a dynamic writable-root sandbox. The collector is non-writing toward the original by contract and mutation detection; completion commands receive write access only in the disposable checkout. Only X/S legs are execution-enforced read-only.
- Claude plan-manager subagents cannot spawn plan-review. Public review therefore stays in main context and uses typed prepare→dispatch→apply; subagents return `NeedsMainReviewDispatch` rather than running the collector inline with writer authority.
- Author company cannot be inferred reliably from the current executor after handoff; persist it when the draft is created and ask once for legacy/unknown plans.
- Ambient model/effort must never be inherited; pins are explicit and dated.
- `timeout 600` is GNU-specific; stock macOS must use the orchestrator's 600-second process/tool timeout and record that evidence.
- A terminal 600-second monotonic deadline is not a transport ETIMEDOUT. Only an execution-layer-typed pre-deadline transport timeout can authorize the one retry.
- Completion review starts only after the `in_review` transition is committed. Capturing HEAD before that commit or reviewing a dirty transition hashes the wrong input.
- New review preserves planned or scheduled; start/fire/auto execution is gated. Missing/stale evidence returns a handback without entering ongoing.

## Global constraints

- Every plan is independently reviewed by default; availability may degrade the number of legs but never silently erases the receipt.
- Cross-company consent (`always | ask | never`) and zero-review progression (`ask | proceed | block`) are independent decisions.
- Both legs are read-only, findings-only, explicit-model, explicit-effort, fresh-context, and consume the same sealed bundle whose canonical plan view is the hashed input.
- Both legs and every receipt echo one request envelope including exact input/bundle/policy hashes; any mismatch is invalid evidence.
- Main-context plan-manager is the sole lifecycle orchestrator and writer; plan-review supplies mechanics/reproduction, and plan-manager subagents only prepare/apply typed envelopes.
- Session-relay is not a schema-v1 transport; the portable CLI path is normative and independent of the relay plugin.
- Host/platform security policy is authoritative and cannot be bypassed by skill text, a user-level receipt, or an alternate transport.
- Skills remain ≤500 lines with current `metadata.updated` + `content_hash`; validators/floors are never weakened.

## STOP conditions

- A reviewer command can write the original repo, inherits ambient permissions/model/effort, reads a moving worktree, or is pointed at a commit that Claude plan mode would need shell access to resolve.
- A live/generated plan-review wrapper retains Edit/Write/writer instructions; a completion runner can write outside its exact disposable checkout; or original tracked/untracked/Git-metadata bytes change between snapshots.
- Cross-company `always` suppresses the zero-review quality decision, or `never` suppresses the same-company leg.
- A receipt reuses after a goal/context/interface/step/acceptance edit, self-invalidates when only its own line/status timestamps change, or lacks exact author/X/S/outcome/time evidence.
- A receipt reuses after any resolved policy/provenance/tier/transport change, or runner/result/leg envelopes disagree on request id, phase, immutable commit/head, input hash, bundle hash, or policy hash.
- Both plan-manager and plan-review can launch/write the same lifecycle review, or lifecycle advances before both legs reach terminal outcomes.
- New/start/schedule/auto changes status before a current eligible receipt, consumes one intent twice, or treats a zero-review `ask` as implicit proceed.
- A manager subagent dispatches another agent, runs the collector inline with writer authority, or prepare/apply request bytes differ.
- Completion capture occurs before the plan-only `in_review` commit, with a dirty scoped worktree, or without the post-run cleanliness recheck.
- A platform denial is retried through another transport, an ambiguous failure is labeled `platform_denied`, or a single-subscription user is blocked despite one successful independent leg.
- The shipped helper imports a consumer dependency, the mutation driver shares implementation/inventory code, the contract/template/docs drift, or a consumer-side setting/env var is added.
