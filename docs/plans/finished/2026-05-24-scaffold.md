# Add scaffold skill (cross-tool, setup + seed modes)

## Goal

One cross-tool `scaffold` skill (`user-invocable: true`) with two modes. **Setup mode** (`scaffold setup`, run once in the source repo) walks the repo and writes `docs/scaffold/spec.yaml` + `templates/` describing the project's structure, conventions, context-tree nodes, bundled skills, and variables. **Seed mode** (`scaffold <target-path>`) reads the spec and generates a new plugin project at the target — root files, context-tree skeleton, bundled skills, manifests — with `{{ var }}` placeholders substituted. Pattern from mattpocock/skills (one config captured once, consumed by a generic skill), folded into a single skill per the user's call. A skill (not a command) so Codex and any agentskills.io runtime can run it — consistent with pipelines-to-skills removing every slash command.

## Context

Every new plugin project repeats the same setup: AGENTS.md/CLAUDE.md pair convention, context-tree nodes, plugin manifests, bridge-skill wiring, starter skills. Repeat work + drift-prone. Scaffolding from a single versioned spec keeps every new project starting at the same baseline.

Depends on:
- foundation-categorization-scoring — spec references category paths (engineering/productivity)
- tree-skill (context-tree) — seed mode writes context-tree-shaped projects; the node convention IS the scaffold output
- pipelines-to-skills — established the commands-removed, skills-only rule this plan follows

### Step details

- **#1 (done)** — Resolved open questions: **one merged skill** (setup + seed modes via argument), **skill-driven `{{ var }}` substitution** (agent fills placeholders as it writes — no sed step in the skill, no templating dep), **bundled** skills (copied from the pinned plugin revision, not fetched), and **refuse `docks`** as a target plugin name.
- **#2** — Single SKILL.md body covers both modes:
  - *setup mode* (`scaffold setup`): walk the repo (reuse context-tree audit to detect nodes), propose the spec, require confirmation, write `docs/scaffold/spec.yaml` + `templates/`.
  - *seed mode* (`scaffold <target-path>`): verify greenfield, load spec, interview for variables (defaults from `git config`), show the full file manifest + resolved values, require explicit confirmation (cross-tool gate — NOT ExitPlanMode), then write everything and `git init` if needed.
  - Mode resolves from the argument; bare `scaffold` prompts which.
  - `references/spec-schema.md` documents the spec.yaml schema (loaded on demand).
- **#3** — Writes in this repo (dogfood — `docs/scaffold/` becomes its own context-tree node):
  ```
  docs/scaffold/
  ├── AGENTS.md      # context-tree node (this folder)
  ├── CLAUDE.md      # @AGENTS.md
  ├── spec.yaml
  └── templates/
      ├── plugin.json.template          # {{ plugin_name }} etc.
      ├── root-AGENTS.md.template
      └── node-templates/{skills,agents,scripts}-AGENTS.md
  ```
- **#5** — Skills aren't shell-callable, so the smoke test exercises the **templates**, not the skill's interactive flow: copy `templates/` → substitute the same `{{ var }}` tokens the skill fills (deterministic sed, fixed test values) → run `bash scripts/ci.sh` in the generated tree → expect green. Because `{{ var }}` substitution is literal find/replace, the sed path and the agent path produce identical output (no mock/prod divergence).

- Network-fetched template versions — bundled only

## Mistakes & Dead Ends

- **2026-05-24** (test scope — a skill isn't shell-invocable): the plan first specced `test-scaffold.sh` to "run the scaffold skill into a temp dir and run that project's ci.sh." → A skill is executed by an agent, not callable from bash, so a shell test can't drive it; and a seeded project that runs docks' FULL ci.sh would need to bundle the entire validator tree + satisfy `claude plugin validate` + ship a `.github/workflows/ci.yml` (heavy, brittle coupling). → Rescoped: `test-scaffold.sh` renders the TEMPLATES with fixed test values into a temp dir and asserts the result is structurally valid (JSON manifests parse, the 3 versioned manifests agree, every context-tree node is a valid pair via `guard-tree.sh <tmp>`, no `{{ }}` leaks). Because `{{ var }}` substitution is literal find/replace, the sed path the test uses and the agent path the skill uses produce identical output. The full self-validating seed is v2 — docks' validators already accept a path arg (only the *default* is `plugins/docks/skills`), so it's a small future step.
- **2026-05-24** (ci.yml drift caught): wiring the scaffold guards into `ci.sh` surfaced that `.github/workflows/ci.yml` never got `guard-tree` from the context-tree ship (tag-CI hadn't run since, so it was an undetected gap). → Fixed in the same pass: added `guard-tree` to the guard job and a `scaffold` job that `pip install pyyaml` then runs both scaffold guards (they skip locally without pyyaml — mirroring ci.sh's YAML check — so tag-CI is where they actually execute).
