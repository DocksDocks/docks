# Fix skill-maintainer timestamps and enforce references/ in /docs

## Goal

Two fixes to the existing skill-maintenance machinery:

1. **Idempotent updates.** The maintainer bumps `metadata.updated:` only when a skill's *meaning* changes (normalized SKILL.md body or normalized references/ content). Re-running on an unchanged skill is a zero-file-write no-op via a stored `content_hash:` in frontmatter.
2. **References/ generation discipline.** `/docs` Phase 3 (Skills Builder) must produce a `references/<topic>.md` split whenever a generated SKILL.md body would exceed 310 lines; Phase 6 (Verifier) hard-fails skills that exceed 310 lines with no references/.

## Context

**Timestamp drift.** Today the maintainer rewrites a SKILL.md and bumps `updated:` even when the rewrite is semantically a no-op (e.g., a sibling context file changed, or a regeneration produced identical output). Every skill always looks "recently updated," so the field can't be used to triage which skills genuinely need a refresh. This compounds against the tree-skill plan's PostToolUse refresh hook — without a semantic-change predicate, the hook will write-loop on every edit.

**References/ gap.** The plugin's own skills follow a strong pattern — SKILL.md body ≤310 lines (sweet spot), depth content split into `references/<topic>.md` (30–150 lines each). The `docs-skills-builder` agent's brief says it should generate references/, but in practice user-project skills generated by `/docs` often end up as monolithic SKILL.md files with no references/. Either the builder doesn't reliably split, or the verifier doesn't catch missing references when SKILL.md exceeds sweet spot. Bring the consumer-side generation up to the standard the plugin itself enforces.

The two fixes ship together because they both touch `/docs` Phase 3/5/6 and the skill-maintenance skill — splitting them would mean two PRs that touch the same files.

### Step details

- **#1** — Predicate table (cheapest-first):
  | Change | Bump `updated:`? | Reason |
  |---|---|---|
  | Frontmatter `description:` text differs | YES | Load-bearing for triggering |
  | Frontmatter `name:` differs | YES | Should never happen, treat as semantic |
  | Frontmatter `allowed-tools:` / `tools:` differs | YES | Capability surface change |
  | Frontmatter `model:` differs | YES | Behavior change |
  | SKILL.md body changes after whitespace normalization | YES | Real content |
  | `references/<name>.md` added/removed | YES | Body-equivalent |
  | Existing `references/<name>.md` body changes after normalization | YES | Real content |
  | Only `metadata.updated:` itself changes | NO | Tautological — write loop |
  | Whitespace-only edits anywhere | NO | Cosmetic |
  | Files outside the skill directory change | NO | Not the skill's content |
- **#3** — "normalize" = strip trailing whitespace per line, collapse blank-line runs, LF line endings. Hash inputs: frontmatter (excluding `updated:` and `content_hash:`) + body + sorted+concatenated `references/*.md` normalized bodies. Output: lowercase hex SHA-256.
- **#4** — Skill-maintenance logic: compute proposed hash → compare to existing `content_hash:` → if equal, skip write entirely (log "no-op (content unchanged)") → if different, write changes + bump `updated:` to today + store new hash.
- **#5** — `--check-only` is what the tree-skill PostToolUse hook calls to decide whether to nudge a refresh; no-op when nothing semantic changed.
- **#7** — Verifier rules after change:
  - body > 500 lines → hard fail (existing)
  - body 310–500 lines AND no references/ → hard fail (NEW)
  - body ≤ 310 lines → no requirement
- **#8** — One-pass backfill computes hash for every existing skill and writes it as frontmatter. After this, the bug is structurally gone.
