# Add scaffold-setup and scaffold skills (cross-tool)

## Goal

Two cross-tool skills (both `user-invocable: true`). `scaffold-setup` is a one-time interactive skill (run in the source repo) that writes `docs/scaffold/spec.yaml` describing the project's structure, conventions, and tree nodes. `scaffold` reads the spec and seeds a new project at a target path — root files, tree skeleton, base skills/agents, manifests. Pattern from mattpocock/skills: a generic skill consumes per-repo config, captured once. Both are skills (not commands) so Codex and any agentskills.io runtime can run them — consistent with the pipelines-to-skills re-architecture that removed every slash command.

## Context

Every new plugin project today repeats the same setup: AGENTS.md/CLAUDE.md pair convention, tree node structure, plugin manifests, bridge skill setup, optional starter skills. Repeat work + drift-prone. Scaffolding from a single versioned spec keeps every new project starting at the same baseline.

Depends on:
- foundation-categorization-scoring — spec references category paths
- tree-skill — `scaffold` writes tree-shaped projects; the tree convention IS the scaffold output
- pipelines-to-skills — established the commands-removed, skills-only rule this plan now follows

### Step details

- **#1** — Proposed spec shape:
  ```yaml
  version: 1
  plugin:
    name_placeholder: "<plugin-name>"
    license: MIT
    manifest_template: plugin.json.template
  tree_nodes:
    - { path: docs/plans, seed_from_skill: plan-init }
    - { path: docs/scaffold, seed: { type: self-reference } }
    - { path: plugins/<plugin>/skills, seed_from_template: tree-templates/skills-AGENTS.md }
  bundled_skills:
    - { source: plugins/docks/skills/productivity/agents,     destination: plugins/<plugin>/skills/productivity/agents }
    - { source: plugins/docks/skills/productivity/tree,       destination: plugins/<plugin>/skills/productivity/tree }
    - { source: plugins/docks/skills/productivity/plan-init,  destination: plugins/<plugin>/skills/productivity/plan-init }
  scripts:
    - { source: scripts/ci.sh }
    - { source: scripts/guard-skills.sh }
    - { source: scripts/score-skills.sh }
    - { source: scripts/scoring.config.json }
    - { source: scripts/guard-tree.sh }
    - { source: scripts/release.sh }
  variables:
    plugin_name:        { prompt: "Plugin name (kebab-case)" }
    plugin_description: { prompt: "Short description" }
    author_name:        { prompt: "Author name", default_from: "git config user.name" }
    author_email:       { prompt: "Author email", default_from: "git config user.email" }
  ```
- **#2** — `scaffold-setup` interviews via prompts (or `AskUserQuestion` where available), walks the current repo to detect tree-node candidates (reuses tree-skill's audit output), shows the proposed spec, requires confirmation before writing.
- **#3** — the `scaffold` skill takes a target path (argument or prompt), verifies greenfield, loads spec, interviews for variables, shows the full file manifest + resolved variable values, and requires explicit user confirmation before writing (cross-tool gate — NOT Plan Mode / `ExitPlanMode`, which are Claude-only). Then writes everything; runs `git init` in target if not a repo.
- **#4** — Creates these files in this repo:
  ```
  docs/scaffold/
  ├── AGENTS.md                  # generated by tree on next refresh
  ├── CLAUDE.md
  ├── spec.yaml
  └── templates/
      ├── plugin.json.template       # with {{ plugin_name }} placeholders
      ├── root-AGENTS.md.template
      └── tree-templates/
          ├── skills-AGENTS.md
          ├── agents-AGENTS.md
          └── scripts-AGENTS.md
  ```
- **#6** — Smoke-test runs the `scaffold` skill into `~/tmp/scaffold-test`, then `bash ~/tmp/scaffold-test/scripts/ci.sh`. The generated project must pass its own CI from a cold start.

## Out of scope

- Generic project generator (Yeoman / cookiecutter equivalent)
- Multi-project sync — `scaffold` is one-way (source → target), no diff-back
- "Apply to existing project" mode — greenfield only this version
- Plugin-name collision detection beyond refusing `docks` as a target name
- Network-fetched template versions — bundled only
