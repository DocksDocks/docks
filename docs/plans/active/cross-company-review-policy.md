# Make cross-company plan review the strong default

## Goal

Promote independent plan review from the current *optional, single, picker-gated* cross-tool second opinion to a **strong, availability-aware default**: every plan drafted with plan-manager is red-teamed, before execution, by **(a) the best available model of the OTHER company** and **(b) a second independent reviewer from the author's company, selected by that company's resolved tier list** — both read-only, findings-only, reconciled by the orchestrator. It is a **strong default, not a hard block** (waivable per-plan; degrades gracefully by availability so a single-subscription user is never blocked), and it runs on a **portable CLI baseline** (`codex exec -s read-only` / `claude -p --permission-mode plan`) with session-relay as an optional async optimization — docks plan-lifecycle must not hard-depend on a separate plugin. Success = the plans contract + its plan-init template + the three skills encode author identity, policy precedence, an ordered model-tier resolver, deterministic degradation/waiver outcomes, canonical draft/completion receipts, sole-writer ownership, and two-reviewer attribution — self-demonstrated by this plan carrying a current-input X receipt plus a successful or explicitly degraded S result.

## Context & rationale

Verbatim user decisions (2026-07-11, via picker + prompt):
- **Enforcement = strong default, availability-aware.** "strong default, but of course when available right? some people don't have double subscription." → NOT a hard block; runs when the model is available; a user with only one company's subscription runs the same-company leg as their independent review, never blocked.
- **Scope = every plan (default posture).** "Every plan, no exceptions." → the review applies to every plan by default; the strong-default waiver + availability degradation (not a stub carve-out) provide proportionality, so a genuinely trivial stub or a single-subscription run naturally executes fewer/zero paid legs.
- **Doctrine home = in the skill, runtime-global-overridable.** "in the skill itself and by global settings it can be overwritten maybe?" → ship model-AGNOSTIC orchestrator doctrine in the skill; already-loaded runtime-global guidance can override the dated default through `ReviewPolicyInput`, without a new consumer config/env contract in this repo.
- **Cross-company consent = standing authorized for this user.** "i always approve cross-review company, dont need to ask that" → a runtime/global `always` authorization suppresses Docks' own consent picker for cross-company review. This standing consent does **not** bypass a host/platform security denial; a denied launch is recorded as a degraded leg with the exact reason, never retried through an alternate export path.
- **Author intent (prompt):** gpt-5.6-sol is the strongest reasoner and avoids Claude's `claude -p --resume` cache-miss on wakes, so it is a sensible default orchestrator run as the *interactive* (cache-warm) session; the orchestrator picks per task between an in-session subagent (this project) and a relay worker (other project/tool/durable). A plan authored by one company's model is reviewed by the other company's best (author gpt-5.6-sol → Claude `fable` high / `opus` xhigh; author Claude → gpt-5.6-sol xhigh) plus a same-company second instance.

Why: the mechanical pieces already exist (one-shot reviewer legs, the red-team pair); this makes independent review the default while staying portable and cost-aware. Cross-company diversity catches blind spots a model family shares; the same-company second instance is the independent leg when only one company is available and a tie-breaker otherwise.

## Environment & how-to-run

- Node 24 + pnpm (`corepack enable && pnpm install --frozen-lockfile`). Validators: `node scripts/ci.mjs`. After any SKILL.md edit, refresh `metadata.updated` + `content_hash` (ci.mjs enforces the hash) and keep bodies ≤500 lines.
- **Portable review baseline (no plugin dependency):** plan-manager first materializes a request bundle under a private temporary directory; both legs read that exact bundle, never a moving worktree or an instruction to run `git show`. Codex = `codex exec -C <bundle> -s read-only -m gpt-5.6-sol -c model_reasoning_effort=xhigh --output-schema <bundle>/reviewer-output.schema.json -- "<fixed rubric + manifest path>"`; Claude = `(cd <bundle> && claude -p --permission-mode plan --model <model> --effort <effort> --json-schema '<same schema>' --output-format json -- "<same rubric + manifest path>")`. This is required because Claude plan mode is read-only and cannot execute `git show`. Never use `--resume`; every leg is fresh. GNU hosts may wrap with `timeout 600`; elsewhere the orchestrator sets a 600-second monotonic tool deadline and records it.
- **Availability probes:** Codex: `command -v codex` then `codex login status`; Claude: `command -v claude` then `claude auth status`. Model availability is attempt-as-probe in the ordered tier list; `unknown model`/entitlement failures fall through once to the next listed model without counting as the leg's retry.
- **Optional relay optimization:** when session-relay is installed, use `relay spawn <bundle> --tool <tool> --model <transport-eligible-model> --effort <effort> --read-only --watch --reply-to <explicit-id> --timeout 600 -- "<task>"`; capture the child id, drain only the explicit mailbox, and accept findings only when the reply echoes child id, request id, input hash, and bundle hash. Bare registration is never completion. The current relay contract forbids Fable as a detached child, so the tier resolver must select the first `relay`-eligible Anthropic tier (currently Opus/max); it must not silently violate the relay skill's model rule.
- **Immutable request bundle:** after committing the candidate, plan-manager fixes `reviewed_commit`, calls the shipped helper to render `plan.review.md` (the exact canonical plan view), and exports every byte-addressed affected path from that commit into `/tmp/docks-plan-review/<request_id>/`. `manifest.json` contains sorted `{path,mode,sha256}` entries, `reviewed_commit`, `input_sha256`, and `bundle_sha256`; the bundle hash covers canonical manifest bytes plus each file's length-prefixed bytes. The helper emits the reviewer-output schema beside it. Plan-manager chmods the bundle read-only before dispatch. Both reviewers read only the bundle. Any missing path, symlink escape, submodule entry, duplicate normalized path, mode/hash mismatch, or bundle mutation STOPs. Accepted findings change canonical input, so plan-manager commits, destroys the old bundle, and repeats both legs; only a no-input-change pass mints a receipt.
- **Product policy vs host policy:** resolve the `ReviewPolicyInput` below from the already-loaded user/runtime-global instruction context; skills do not read a new env var or consumer config file. Host sandbox/export policy remains an outer gate. An explicit host denial is `platform_denied`; it is never retried through another transport. An ambiguous launch failure is `unavailable_unknown`, never guessed to be policy denial.

## Interfaces & data shapes

- **Author identity (new base plan frontmatter, captured at draft creation):** `review_author_company: openai | anthropic | unknown`, `review_author_tool: <string>`, `review_author_model: <string>`, `review_author_effort: <string>`. A legacy/unknown company triggers one picker before first review and is then persisted; X = other company, S = an independent reviewer selected from the persisted company's resolved tier list (not necessarily the exact author model). This plan records the originating Claude session as `anthropic/claude/unknown/unknown`; gpt-5.6-sol is the later orchestrator, not the author.
- **Resolved policy input (logical input supplied by the orchestrator after normal instruction precedence):** `ResolvedReviewPolicy = { schema: 1, cross_company_consent: always | ask | never, zero_reviewer_policy: ask | proceed | block, orchestrator_preference: auto | in_session | cli | relay, openai_tiers: [{ model, effort, transports: [in_session | cli | relay] }], anthropic_tiers: [{ model, effort, transports: [...] }], provenance: { <field>: current_user | runtime_global | skill_default } }`. Defaults are expanded before the shipped helper JCS-encodes and hashes the policy as `policy_sha256`; tier and transport order are semantic. Transport selection happens once before attempts; an authoritative denial never switches transport. The runner request/result/receipts carry the exact snapshot+hash, and reuse re-resolves current instructions and matches both. `always` suppresses only X export consent, never zero-review quality. Current-turn user instruction overrides runtime-global guidance, which overrides skill defaults.
- **Ordered model-tier resolver (dated 2026-07; skill text says "check the current tier list"):**
  - OpenAI best: `gpt-5.6-sol` effort `xhigh`, eligible for `[in_session,cli,relay]`.
  - Claude best: `fable` effort `high`, eligible for `[in_session,cli]`; then `opus` effort `max`, eligible for `[in_session,cli,relay]`. A relay leg therefore begins with Opus/max under the current relay contract.
- **Leg + attempt result enums and ordered classifier:** `LegResult = passed | waived | not_authorized | unavailable_auth | unavailable_model | timed_out | platform_denied | failed_unparseable | unavailable_unknown`; `AttemptResult = passed | auth_failed | model_unavailable | deadline_exceeded | platform_denied | transient_transport | nonzero_exit | signaled | unparseable`. Classify in order: exact matching waiver → `waived`; product consent denies X → `not_authorized`; authoritative policy denial → `platform_denied`; auth preflight fails → `unavailable_auth`; every transport-eligible tier attempt is `model_unavailable` → `unavailable_model`; monotonic deadline → `timed_out`; exit 0 with schema-invalid output → `failed_unparseable`; success + schema-valid output → `passed`; every other terminal sequence → `unavailable_unknown`. One transient retry exists **per leg, not per tier**, and only repeats the same model/transport after pre-output typed `EAGAIN`, `ETIMEDOUT`, or `ECONNRESET`. Thus attempts are ordered and bounded `0..eligible_tier_count+1`; the extra slot is used at most once. Model-unavailable fallthrough consumes an attempt but not the retry. Strings, output-started failures, deadline expiry, and other results never retry.
- **Two-reviewer attribution grammar (extends `docs/plans/AGENTS.md`), leg-namespaced IDs, reasons preserved:**
  ```markdown
  Cross-check (<date>): [X: <company-B> <model> <effort>] N findings (sev) — accepted X<ids> / rejected X<ids> (one-line reason each); [S: <company-A second> <model> <effort>] M findings — accepted S<ids> / rejected S<ids> (reason each); [<author>] independently verified <X/S ids> against source before accepting.
  DISAGREEMENT: <topic> — [X<id>] <pos> / [S<id>] <pos>. Kept: <choice> — decided by <orchestrator|user via picker>, because <one line>.
  ```
- **Availability-degradation matrix (strong default, never a subscription hard block):** both passed → dual review; one passed → record the other exact outcome and proceed; zero passed → apply `zero_reviewer_policy`. Draft phase: `ask` picker, `proceed` writes a degraded receipt and remains eligible to start, `block` writes a blocked Self-review receipt and remains `planned`. Completion phase: the committed plan remains `in_review`; `ask` picker, `proceed` writes degraded completion evidence/verdict, `block` writes a blocked Review receipt with `review_status: null`, cannot ship, and retries the same clean request only while input/policy remain exact. No completion outcome rolls back to `planned`. `cross_company_consent=never` records X as `not_authorized` and never suppresses S.
- **Prompted decision evidence:** `ReviewDecisionEvidence = { schema: 1, kind: x_consent | zero_reviewer, decision: allow | deny | proceed | block, actor: <non-empty string>, reason: <non-empty string>, at: <ISO datetime>, request_id: <uuid>, input_sha256: <64hex> }`. `cross_company_consent=ask` requires an `x_consent` decision before an X attempt; zero-passed plus `zero_reviewer_policy=ask` requires `zero_reviewer` evidence before the phase outcome. The decision is echoed in the run result and receipt. Standing `always`/`never` and configured `proceed`/`block` use the resolved policy+provenance and require `decision_evidence:null`; a prompted outcome with null or mismatched evidence STOPs.
- **Per-plan waivers (new frontmatter):** `ReviewWaiver = { phase: draft | completion, input_sha256: <64hex>, legs: [X | S], actor: <string>, reason: <non-empty string>, at: <ISO datetime> }`; `review_waivers: [ReviewWaiver]`. plan-manager writes one only from an explicit current-user decision; consent `never` is not a waiver. The shipped helper normalizes duplicate-free legs to `[X,S]`, JCS-encodes, and hashes the waiver. Uniqueness key `(phase,input_sha256,leg)` permits exactly one match; duplicate/conflicting entries STOP without a receipt. Every waived leg embeds the normalized snapshot/hash and reuse requires the exact persisted unique waiver; removal, duplication, reassignment, or field mutation invalidates it.
- **Immutable request envelope:** `ReviewRequestEnvelope = { schema: 1, request_id: <uuid>, phase: draft | completion, lifecycle_intent: none | start | schedule_fire | auto_execute, reviewed_commit_or_head: <40hex>, input_sha256: <64hex>, bundle_sha256: <64hex>, policy: ResolvedReviewPolicy, policy_sha256: <64hex> }`. One envelope is created only after the plan transition/draft input is committed and the bundle is sealed; both legs, every result, and every persisted leg echo it byte-exactly. A mismatched/stale envelope or bundle is a schema failure, not a review outcome.
- **Attempt + raw leg evidence (closed objects; unknown keys reject):** `AttemptEvidence = { schema: 1, model: string, effort: string, transport: in_session | cli | relay, started: bool, output_started: bool, result: AttemptResult, exit_code: i32 | null, signal: string | null, child_id: string | null, denial_source: sandbox | managed_policy | runtime_policy | null, retry_cause: transport_EAGAIN | transport_ETIMEDOUT | transport_ECONNRESET | null, timeout_mode: gnu_timeout | orchestrator_tool | null, timeout_seconds: 600, reason: string, stdout_sha256: 64hex | null, stderr_sha256: 64hex | null }`. `RawLegEvidence = { schema: 1, leg: X | S, request: ReviewRequestEnvelope, result: LegResult, attempts: [AttemptEvidence], selected: {model:string,effort:string,transport:...} | null, findings: [ReviewerFinding], findings_sha256: 64hex | null, severity_totals: {high:u32,medium:u32,low:u32}, waiver: ReviewWaiver | null, decision_evidence: ReviewDecisionEvidence | null, reason: string | null }`. Passed requires selected, last attempt passed, schema-valid findings/hash/totals, null waiver/reason, and the attempt bound above. Waived requires exact waiver and zero attempts/findings. Not-authorized requires X, zero attempts/findings, null selected/waiver/hash, and policy or prompted deny evidence. Other terminal results require non-empty reason, zero findings, null selected/waiver/hash, and exact terminal attempt evidence when started. Free-form stderr never proves policy denial.
- **Reconciliation evidence:** the shipped helper sorts findings by unique leg-prefixed ID (`X1…`/`S1…`), JCS-encodes the list, and defines `findings_sha256`. `ReconciliationEvidence = { accepted: [<id>], rejected: [{ id, reason }] }` is produced only by plan-manager after independent validation; sets are disjoint/duplicate-free, every ID exists, every reproduced finding appears exactly once, and rejection reasons are non-empty. A persisted leg is `{ request, raw, reconciliation }`; plan-review never constructs reconciliation.
- **Versioned reviewer output and run results:** `ReviewerFinding = { id: X<u32> | S<u32>, severity: high | medium | low, section: string, path: string | null, locator: string | null, defect: string, fix: string, evidence: string }`; `ReviewerOutput = { schema: 1, leg: X | S, request_id: uuid, input_sha256: 64hex, bundle_sha256: 64hex, verdict: ready | not_ready, score: u8_0_100, findings: [ReviewerFinding], confirmations: [string] }`. The helper exports one JSON Schema with `additionalProperties:false`; Codex receives it via `--output-schema`, Claude via `--json-schema`. `FindingEvidence = { id: string, source: X | S | primary, severity: high | medium | low, path: string | null, locator: string | null, defect: string, fix: string, reproduction: { method: read | command, command: string | null, exit_code: i32 | null, evidence_sha256: 64hex } }`. `AcceptanceEvidence = { criterion_id:string, command:string, expected:string, exit_code:i32, actual_sha256:64hex, met:bool }`; `CiEvidence = { command:string, exit_code:i32, first_failure:string | null, output_sha256:64hex }`; `PrimaryCompletionEvidence = { goal_met: yes | partial | no, findings:[FindingEvidence], acceptance:[AcceptanceEvidence], ci:CiEvidence, regressions:[string], followups:[string] }`. `DraftRunResult = { schema:1, kind:draft, request, X:RawLegEvidence, S:RawLegEvidence, reproduced:[FindingEvidence], decision_evidence:ReviewDecisionEvidence | null, outcome:dual | single | zero_degraded | blocked, pre_execution_eligible:bool }`. `CompletionRunResult = { schema:1, kind:completion, request, plan_input_sha256:64hex, diff_sha256:64hex, X, S, reproduced, decision_evidence, outcome, primary:PrimaryCompletionEvidence }`. All are closed discriminated objects; missing, extra, mistyped, or cross-variant fields reject.
- **Dependency-free shipped canonicalizer/validator:** `plugins/docks/skills/productivity/plan-review/scripts/review-policy.mjs` is the sole consumer implementation, runs on Node 20+, and imports only `node:` built-ins—never the author repo's `yaml` devDependency or consumer `node_modules`. It parses the deliberately closed plans-frontmatter grammar: top-level string keys; quoted/plain/null/boolean/safe-integer scalars; flow scalar arrays used by tags; indented scalar block arrays used by paths; and `review_waivers` as one-line strict JSON/JCS array. Nested YAML maps, arbitrary YAML objects, floats, aliases, tags, duplicate keys, non-string keys, tabs, lone surrogates, and out-of-range integers reject. The plan contract/template/manager must emit only this grammar. Canonical JSON follows RFC 8785 JCS; strings are not normalized and property order is UTF-16 code-unit order. The helper owns the plan-view renderer, bundle manifest/hash, policy/waiver/finding/request/result validation, reviewer JSON Schema, receipt validation, and SHA-256. Author tests execute that exact helper; a temp consumer containing only the plugin payload and no root package manifest/node_modules must pass.
- **Canonical draft view + receipt:** the helper rejects BOM/invalid UTF-8/CR-only input and normalizes CRLF→LF. It removes exactly these frontmatter keys from the review view: `updated`, `status`, `started_at`, `in_review_since`, `blocked_reason`, `blocked_since`, `assignee`, `review_status`, `ship_commit`, and `review_waivers`; the canonical waiver hash is bound separately in each waived leg. No other key is mutable/excluded. In the body it retains all Self-review/Review prose and removes only exact outside-fence machine lines beginning `Review-receipt: ` or `Completion-review-receipt: `; duplicates or multiline receipt records reject. It emits `plan.review.md` as `frontmatter-jcs + "\n" + retained-body-with-one-terminal-LF`; reviewers consume exactly these bytes and `input_sha256` hashes exactly them. Store compact JCS inside `## Self-review`: `Review-receipt: {"schema":1,"phase":"draft","request":<ReviewRequestEnvelope>,"input_sha256":"<64hex>","reviewed_commit":"<40hex>","author":{"company":"...","tool":"...","model":"...","effort":"..."},"policy":<ResolvedReviewPolicy>,"policy_sha256":"<64hex>","X":<persisted-leg>,"S":<persisted-leg>,"decision_evidence":<ReviewDecisionEvidence|null>,"outcome":"dual|single|zero-degraded|blocked","reviewed_at":"<ISO>"}`. Any non-receipt Self-review/Review edit invalidates reuse; only the enumerated lifecycle fields, waiver field with separately matched hash, and machine receipt lines do not.
- **Public entry, pre-execution gate, and sole-writer protocol:** `new plan`, `review plan`, `start`, scheduled fire, and `auto_execute` all route through main-context plan-manager. One typed protocol owns them: `prepare(intent)` commits/captures input, seals the bundle, and returns `NeedsMainReviewDispatch { request }`; main dispatches plan-review exactly once; `apply(request,result)` re-resolves policy, re-hashes input/bundle, validates/reconciles/writes the receipt, then applies the same request's `lifecycle_intent` only when `pre_execution_eligible=true`. Missing/stale/blocked receipts STOP before `status: ongoing`; they never perform a transition and promise a later review. `start`/fire with no current receipt calls `prepare` rather than mutating. Zero-review block remains `planned`; noninteractive `ask` remains planned and surfaces a pending decision. A schedule/auto intent is consumed exactly once under the input/policy hash. Manager wrappers expose only prepare/apply and return the handback; they never dispatch children. plan-review is internal, `user-invocable:false`, and evidence-only. It launches read-only X/S legs and returns the closed run result; no other workflow may launch a pair for the same phase/hash.
- **Canonical completion input + disposable verification:** exact order is: mark steps done → commit plan-only `status: in_review`/`in_review_since` → assert scoped clean → capture `reviewed_head`, canonical plan input, policy/hash, and canonical binary diff → create a detached disposable checkout at `reviewed_head` outside the original repo → invoke plan-review with the original/bundle read-only and workspace-write confined to the disposable checkout → run acceptance/CI and reproduce command findings there → destroy the disposable checkout → re-hash original tracked blobs/modes, untracked path+content, `.git/config`, HEAD/index/refs, and scoped cleanliness → write verdict/receipt → commit the plan-only review edit. CI may create artifacts only in the disposable checkout; any original mutation STOPs. Pathspecs are bytewise UTF-8 sorted. Diff bytes are `LC_ALL=C git -c diff.renames=false diff --no-ext-diff --no-textconv --no-color --binary <planned_at_commit>..<reviewed_head> -- <sorted-pathspecs>`; nonzero or filters that alter no-textconv STOP. Receipt: `Completion-review-receipt: {"schema":1,"phase":"completion","request":<ReviewRequestEnvelope>,"planned_at_commit":"<sha>","reviewed_head":"<sha>","diff_sha256":"<hex>","plan_input_sha256":"<hex>","policy":<ResolvedReviewPolicy>,"policy_sha256":"<hex>","X":<persisted-leg>,"S":<persisted-leg>,"decision_evidence":<ReviewDecisionEvidence|null>,"primary":<PrimaryCompletionEvidence>,"outcome":"dual|single|zero-degraded|blocked","reviewed_at":"<ISO>"}`. No self-referential review commit exists. Ship-time reuse allows only the later plan-only receipt commit and requires canonical input, diff, original cleanliness, and policy unchanged.

## Known gotchas

- Relay `spawn` returns at birth-registration, BEFORE findings exist — never advance the lifecycle on a bare spawn; require `--watch`+bus-reply or use the synchronous CLI leg.
- Claude `--permission-mode plan` cannot execute `git show`; immutable review therefore depends on the sealed bundle, not a commit named only in prose.
- The relay skill currently forbids Fable detached children even though Fable is the first CLI/in-session Anthropic tier; transport eligibility must select Opus/max for relay.
- Two reviewer lists must namespace finding IDs (X#/S#) and preserve per-finding accept/reject reasons, or reconciliation is ambiguous.
- A receipt stored inside the plan can invalidate a naïve whole-file hash. Use the canonical exclusion algorithm exactly; any substantive section change still invalidates it.
- Product consent and host sandbox policy are different signals. `always` is not a sandbox override, and host denial is not permission to try a second export transport.
- A standing consent or zero-review decision is part of the resolved policy hash. A receipt minted under yesterday's policy cannot authorize today's run after any current-user/runtime-global/default tier or provenance change.
- Plugin-shipped Claude agents cannot enforce a dynamic writable-root sandbox. The collector is non-writing toward the original by contract and mutation detection; completion commands receive write access only in the disposable checkout. Only X/S legs are execution-enforced read-only.
- Claude plan-manager subagents cannot spawn plan-review. Public review therefore stays in main context and uses typed prepare→dispatch→apply; subagents return `NeedsMainReviewDispatch` rather than running the collector inline with writer authority.
- Author company cannot be inferred reliably from the current executor after handoff; persist it when the draft is created and ask once for legacy/unknown plans.
- Ambient model/effort must never be inherited; pins are explicit and dated.
- `timeout 600` is GNU-specific; stock macOS must use the orchestrator's 600-second process/tool timeout and record that evidence.
- A terminal 600-second monotonic deadline is not a transport ETIMEDOUT. Only an execution-layer-typed pre-deadline transport timeout can authorize the one retry.
- Completion review starts only after the `in_review` transition is committed. Capturing HEAD before that commit or reviewing a dirty transition hashes the wrong input.
- New/start/schedule/auto execution is itself gated: a missing/stale receipt returns a handback while the plan remains planned. A promise to review after entering ongoing is a policy violation.

## Global constraints

- Every plan is independently reviewed by default; availability may degrade the number of legs but never silently erases the receipt.
- Cross-company consent (`always | ask | never`) and zero-review progression (`ask | proceed | block`) are independent decisions.
- Both legs are read-only, findings-only, explicit-model, explicit-effort, fresh-context, and consume the same sealed bundle whose canonical plan view is the hashed input.
- Both legs and every receipt echo one request envelope including exact input/bundle/policy hashes; any mismatch is invalid evidence.
- Main-context plan-manager is the sole lifecycle orchestrator and writer; plan-review supplies mechanics/reproduction, and plan-manager subagents only prepare/apply typed envelopes.
- Session-relay is optional; the portable CLI path is normative.
- Host/platform security policy is authoritative and cannot be bypassed by skill text, a user-level receipt, or an alternate transport.
- Skills remain ≤500 lines with current `metadata.updated` + `content_hash`; validators/floors are never weakened.

## STOP conditions

- A reviewer command can write the original repo, inherits ambient permissions/model/effort, reads a moving worktree, or is pointed at a commit that Claude plan mode would need shell access to resolve.
- A live/generated plan-review wrapper retains Edit/Write/writer instructions; a completion runner can write outside its exact disposable checkout; or original tracked/untracked/Git-metadata bytes change between snapshots.
- Cross-company `always` suppresses the zero-review quality decision, or `never` suppresses the same-company leg.
- A receipt reuses after a goal/context/interface/step/acceptance edit, self-invalidates when only its own line/status timestamps change, or lacks exact author/X/S/outcome/time evidence.
- A receipt reuses after any resolved policy/provenance/tier/transport change, or runner/result/leg envelopes disagree on request id, phase, immutable commit/head, input hash, bundle hash, or policy hash.
- Both plan-manager and plan-review can launch/write the same lifecycle review, or lifecycle advances before both legs reach terminal outcomes.
- New/start/schedule/auto changes status before a current eligible receipt, consumes one intent twice, or treats a zero-review `ask` as implicit proceed.
- A manager subagent dispatches another agent, runs the collector inline with writer authority, or prepare/apply request bytes differ.
- Completion capture occurs before the plan-only `in_review` commit, with a dirty scoped worktree, or without the post-run cleanliness recheck.
- A platform denial is retried through another transport, an ambiguous failure is labeled `platform_denied`, or a single-subscription user is blocked despite one successful independent leg.
- The shipped helper imports a consumer dependency, the mutation driver shares implementation/inventory code, the contract/template/docs drift, or a consumer-side setting/env var is added.
