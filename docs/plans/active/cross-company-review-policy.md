# Make cross-company plan review the strong default

## Goal

Promote independent plan review from the current *optional, single, picker-gated* cross-tool second opinion to a **strong, availability-aware default**: every plan drafted with plan-manager is red-teamed, before execution, by **(a) the best available model of the OTHER company** and **(b) a second independent reviewer from the author's company, selected by that company's resolved tier list** — both read-only, findings-only, reconciled by the orchestrator. It is a **strong default, not a hard block** (waivable per-plan; degrades gracefully by availability so a single-subscription user is never blocked), and it runs on a **portable CLI baseline** (`codex exec -s read-only` / `claude -p --permission-mode plan`) with session-relay as an optional async optimization — docks plan-lifecycle must not hard-depend on a separate plugin. Success = the plans contract + its plan-init template + the three skills encode author identity, policy precedence, an ordered model-tier resolver, deterministic degradation/waiver outcomes, canonical draft/completion receipts, sole-writer ownership, and two-reviewer attribution — self-demonstrated by this plan carrying a current-input X receipt plus a successful or explicitly degraded S result.

## Context & rationale

Verbatim user decisions (2026-07-11, via picker + prompt):
- **Enforcement = strong default, availability-aware.** "strong default, but of course when available right? some people don't have double subscription." → NOT a hard block; runs when the model is available; a user with only one company's subscription runs the same-company leg as their independent review, never blocked.
- **Scope = every plan (default posture).** "Every plan, no exceptions." → the review applies to every plan by default; the strong-default waiver + availability degradation (not a stub carve-out) provide proportionality, so a genuinely trivial stub or a single-subscription run naturally executes fewer/zero paid legs.
- **Doctrine home = in the skill, runtime-global-overridable.** "in the skill itself and by global settings it can be overwritten maybe?" → ship model-AGNOSTIC orchestrator doctrine in the skill; already-loaded runtime-global guidance can override the dated default through `ReviewPolicyInput`, without a new consumer config/env contract in this repo.
- **Cross-company consent = standing authorized for this user.** "i always approve cross-review company, dont need to ask that" → a runtime/global `always` authorization suppresses Docks' own consent picker for cross-company review. This standing consent does **not** bypass a host/platform security denial; a denied launch is recorded as a degraded leg with the exact reason, never retried through an alternate export path.
- **Author intent (prompt):** gpt-5.6-sol is the strongest reasoner and avoids Claude's `claude -p --resume` cache-miss on wakes, so it is a sensible default orchestrator run as the *interactive* (cache-warm) session; the orchestrator picks per task between an in-session subagent (this project) and a relay worker (other project/tool/durable). A plan authored by one company's model is reviewed by the other company's best (author gpt-5.6-sol → Claude `fable` high / `opus` xhigh; author Claude → gpt-5.6-sol xhigh) plus a same-company second instance.

Why: the mechanical pieces already exist (one-shot reviewer legs, the red-team pair); this makes independent review the default while staying portable and cost-aware. Cross-company diversity catches blind spots a model family shares; the same-company second instance is the independent leg when only one company is available and a tie-breaker otherwise.

## Environment & how-to-run

- Node 24 + pnpm (`corepack enable && pnpm install --frozen-lockfile`). Validators: `node scripts/ci.mjs`. After any SKILL.md edit, refresh `metadata.updated` + `content_hash` (ci.mjs enforces the hash) and keep bodies ≤500 lines.
- **Portable review baseline (no plugin dependency):** Codex = `codex exec -s read-only -m gpt-5.6-sol -c model_reasoning_effort=xhigh -- "<fixed rubric + immutable plan commit/path>"`; Claude = `claude -p --permission-mode plan --model <fable|opus> --effort <high|xhigh> --output-format json -- "<same fixed rubric + immutable plan commit/path>"`. Never use `--resume`; every leg is a fresh context. GNU hosts may wrap with `timeout 600`; on hosts without GNU `timeout`, the orchestrating runtime must set its process/tool timeout to 600 seconds and record that enforcement in the receipt.
- **Availability probes:** Codex: `command -v codex` then `codex login status`; Claude: `command -v claude` then `claude auth status`. Model availability is attempt-as-probe in the ordered tier list; `unknown model`/entitlement failures fall through once to the next listed model without counting as the leg's retry.
- **Optional relay optimization:** when session-relay is installed, use `relay spawn <dir> --tool <tool> --model <model> --effort <effort> --read-only --watch --reply-to <explicit-id> --timeout 600 -- "<task>"`; capture the returned child id, drain only the explicit reply mailbox, and accept findings only when the message names that child id plus the immutable review-input hash. Bare birth registration is never completion.
- **Immutable draft order:** plan-manager first writes and auto-commits the candidate draft; `reviewed_commit=$(git rev-parse HEAD)` is then fixed for both legs. Reviewers read the plan and cited affected paths from that commit (`git show <reviewed_commit>:<path>`), never a moving worktree. If accepted findings change canonical review input, plan-manager commits the revision and repeats both legs; only a no-input-change pass can mint the reusable receipt.
- **Product policy vs host policy:** resolve the `ReviewPolicyInput` below from the already-loaded user/runtime-global instruction context; skills do not read a new env var or consumer config file. Host sandbox/export policy remains an outer gate. An explicit host denial is `platform_denied`; it is never retried through another transport. An ambiguous launch failure is `unavailable_unknown`, never guessed to be policy denial.

## Interfaces & data shapes

- **Author identity (new base plan frontmatter, captured at draft creation):** `review_author_company: openai | anthropic | unknown`, `review_author_tool: <string>`, `review_author_model: <string>`, `review_author_effort: <string>`. A legacy/unknown company triggers one picker before first review and is then persisted; X = other company, S = an independent reviewer selected from the persisted company's resolved tier list (not necessarily the exact author model). This plan records the originating Claude session as `anthropic/claude/unknown/unknown`; gpt-5.6-sol is the later orchestrator, not the author.
- **Resolved policy input (logical input supplied by the orchestrator after normal instruction precedence):** `ResolvedReviewPolicy = { schema: 1, cross_company_consent: always | ask | never, zero_reviewer_policy: ask | proceed | block, orchestrator_preference: auto | in_session | relay, openai_tiers: [{ model, effort }], anthropic_tiers: [{ model, effort }], provenance: { <field>: current_user | runtime_global | skill_default } }`. Defaults are `ask`, `ask`, `auto`, and the dated tier list below; the resolved object always expands defaults and optionals before hashing. Provenance is per field because mixed precedence is valid. Canonical policy JSON recursively sorts object keys lexicographically, preserves tier-array order, encodes UTF-8 compact JSON, and hashes those exact bytes as `policy_sha256`. The runner request, result, and both receipts carry the exact snapshot plus hash; receipt reuse requires re-resolving current instructions and matching both bytes and hash. `cross_company_consent=always` suppresses only the X-leg export-consent picker; it never authorizes proceeding with zero reviewers. A current-turn user instruction overrides runtime-global guidance, which overrides skill defaults.
- **Ordered model-tier resolver (dated 2026-07; skill text says "check the current tier list"):**
  - OpenAI best: `gpt-5.6-sol` effort `xhigh`.
  - Claude best: try `fable` effort `high`; if unavailable, `opus` effort `xhigh`. (Deterministic order + per-model availability probe; fall through on probe failure.)
- **Leg result enum + ordered classifier:** `passed | waived | not_authorized | unavailable_auth | unavailable_model | timed_out | platform_denied | failed_unparseable | unavailable_unknown`. Classify in order: exact matching per-plan waiver → `waived`; product consent denies X → `not_authorized`; an authoritative host/runtime policy denial at any stage → `platform_denied` (`started` remains evidence, not a different class); auth preflight fails → `unavailable_auth`; every ordered model returns explicit unknown/entitlement failure → `unavailable_model`; the orchestrator's one monotonic 600-second deadline fires → terminal `timed_out`, never retried; exit 0 but findings cannot be parsed → `failed_unparseable`; success + parse → `passed`; every other nonzero exit, signal, retry exhaustion, or ambiguous failure → `unavailable_unknown`. One same-model/same-transport retry is allowed only when the execution layer explicitly types a pre-output failure as transient transport `EAGAIN`, transport `ETIMEDOUT`, or transport `ECONNRESET` **before** the monotonic deadline; an errno/string without transport provenance, any parseable output, or deadline expiry is non-retryable, and no retry may switch transport.
- **Two-reviewer attribution grammar (extends `docs/plans/AGENTS.md`), leg-namespaced IDs, reasons preserved:**
  ```markdown
  Cross-check (<date>): [X: <company-B> <model> <effort>] N findings (sev) — accepted X<ids> / rejected X<ids> (one-line reason each); [S: <company-A second> <model> <effort>] M findings — accepted S<ids> / rejected S<ids> (reason each); [<author>] independently verified <X/S ids> against source before accepting.
  DISAGREEMENT: <topic> — [X<id>] <pos> / [S<id>] <pos>. Kept: <choice> — decided by <orchestrator|user via picker>, because <one line>.
  ```
- **Availability-degradation matrix (strong default, never a subscription hard block):** both passed → dual review; one passed → record the other exact outcome and proceed; zero passed → apply `zero_reviewer_policy` (`ask` = picker, `proceed` = surfaced degraded receipt, `block` = remain planned). `cross_company_consent=never` records X as `not_authorized` and skips its launch; it never suppresses S. A single-subscription user therefore proceeds with the available independent leg, while a host denial is visible and never bypassed.
- **Per-plan waivers (new frontmatter):** `ReviewWaiver = { phase: draft | completion, input_sha256: <64hex>, legs: [X | S], actor: <string>, reason: <non-empty string>, at: <ISO datetime> }`; `review_waivers: [ReviewWaiver]`. plan-manager writes one only from an explicit current-user decision; `cross_company_consent=never` is `not_authorized`, never a waiver. A waiver cannot cross phase/input. Canonical waiver JSON recursively sorts every object key lexicographically, normalizes `legs` to a duplicate-free `[X,S]` order, encodes UTF-8 compact JSON with no insignificant whitespace, and hashes those exact bytes with SHA-256. The uniqueness key is `(phase,input_sha256,leg)`: more than one matching entry for a leg, including semantically identical duplicates or conflicting entries, is a STOP with no receipt. Canonical plan content excludes the waiver list, but every waived leg embeds `waiver: { snapshot: <exact normalized ReviewWaiver>, sha256: <canonical-json-sha256> }`; receipt reuse requires every snapshot field and hash to match the one currently persisted unique waiver. Removal, duplication, leg reassignment, or actor/reason/time mutation invalidates reuse.
- **Immutable request envelope:** `ReviewRequestEnvelope = { schema: 1, request_id: <uuid>, phase: draft | completion, reviewed_commit_or_head: <40hex>, input_sha256: <64hex>, policy: ResolvedReviewPolicy, policy_sha256: <64hex> }`. One envelope is created after the plan transition/draft is committed; both legs, every raw result variant, and every persisted leg echo it byte-exactly. A mismatched/stale envelope is a schema failure, not a review outcome.
- **Attempt + raw leg evidence:** `AttemptEvidence = { model, effort, transport: cli | relay, started: bool, result, exit_code: i32 | null, signal: string | null, denial_source: sandbox | managed_policy | runtime_policy | null, retry_cause: typed_transient | transport_EAGAIN | transport_ETIMEDOUT | transport_ECONNRESET | null, timeout_mode: gnu_timeout | orchestrator_tool | null, timeout_seconds: 600, reason: string, stdout_sha256: string | null, stderr_sha256: string | null }`. `RawLegEvidence = { request: ReviewRequestEnvelope, result, model: string | null, effort: string | null, findings: u32, severity: { high: u32, medium: u32, low: u32 }, reason, child_id: string | null, attempts: [AttemptEvidence], findings_sha256: string | null, waiver: null | { snapshot: ReviewWaiver, sha256: string } }`. Attempt order and typed fields prove tier fallback, retry, timeout, and policy-denial behavior. An authoritative policy denial is a typed execution-layer signal at any launch stage; free-form stderr alone never proves `platform_denied`.
- **Reconciliation evidence:** canonical finding-list JSON sorts findings by unique leg-prefixed ID (`X1…`/`S1…`), recursively sorts object keys, preserves nested evidence arrays, emits UTF-8 compact JSON, and defines `findings_sha256`. `ReconciliationEvidence = { accepted: [<id>], rejected: [{ id, reason }] }` is produced only by plan-manager after it independently validates the runner's reproduced findings; IDs are unique, accepted/rejected are disjoint and duplicate-free, every ID exists in that leg's canonical finding list, every reproduced finding appears in exactly one set, and every rejection reason is non-empty. A persisted leg is `PersistedLegEvidence = { request: ReviewRequestEnvelope, raw: RawLegEvidence, reconciliation: ReconciliationEvidence }`; plan-review never constructs accepted/rejected positions.
- **Review evidence types:** `FindingEvidence = { id: string, source: X | S | primary, severity: high | medium | low, path: string | null, line: u32 | null, defect: string, fix: string, reproduction: { method: read | command, command: string | null, exit_code: i32 | null, evidence_sha256: string } }`. `AcceptanceEvidence = { criterion_id: string, command: string, expected: string, exit_code: i32, actual_sha256: string, met: bool }`. `CiEvidence = { command: string, exit_code: i32, first_failure: string | null, output_sha256: string }`.
- **Canonical draft input + receipt:** the normative harness algorithm is: reject BOM/invalid UTF-8/CR-only input; normalize CRLF→LF; parse exactly one `---` YAML frontmatter document with aliases/custom tags and duplicate keys rejected; remove mutable keys (`status`, `updated`, `started_at`, `assignee`, `review_status`, `in_review_since`, `ship_commit`, `review_waivers`); recursively key-sort the resulting YAML value and encode it as compact JSON. Scan the body linewise with CommonMark backtick/tilde fence state; outside fences, require at most one exact column-0 `## Self-review` and one `## Review`, remove each through the next outside-fence column-0 level-2 heading or EOF, and reject duplicates. Hash bytes `canonical-frontmatter-json + "\n" + retained-body-with-exactly-one-terminal-LF`. Golden vectors cover multiline YAML, duplicate keys/sections, fenced headings, CRLF, lifecycle-only stability, and substantive invalidation. Store one compact canonical JSON line inside `## Self-review`: `Review-receipt: {"schema":1,"phase":"draft","request":<ReviewRequestEnvelope>,"input_sha256":"<64hex>","reviewed_commit":"<40hex>","author":{"company":"...","tool":"...","model":"...","effort":"..."},"policy":<ResolvedReviewPolicy>,"policy_sha256":"<64hex>","X":<persisted-leg>,"S":<persisted-leg>,"outcome":"dual|single|zero-degraded|blocked","reviewed_at":"<ISO>"}`. Waiver/receipt/lifecycle metadata are excluded, so ingestion does not self-invalidate; any substantive input or currently resolved policy change does.
- **Public entry + dispatch ownership protocol:** `review plan <slug>` is a plan-manager operation; plan-review is an internal, `user-invocable: false`, evidence-only runner. plan-manager is the sole lifecycle orchestrator and writer. It calls one plan-review runner exactly once with the `ReviewRequestEnvelope`; plan-review alone spawns/collects reviewers **and reproduces every finding**, returning one discriminated variant without editing: `DraftRunResult = { status: complete, request: ReviewRequestEnvelope, X: RawLegEvidence, S: RawLegEvidence, findings: [FindingEvidence] } | { status: blocked, request: ReviewRequestEnvelope, reason: string, X: RawLegEvidence | null, S: RawLegEvidence | null, findings: [FindingEvidence] }`; `CompletionRunResult = { status: complete, request: ReviewRequestEnvelope, X: RawLegEvidence, S: RawLegEvidence, findings: [FindingEvidence], acceptance_evidence: [AcceptanceEvidence], ci: CiEvidence, regressions: [FindingEvidence], goal_met: yes | partial | no, review_status: passed | partial | regressed, receipt_inputs: { planned_at_commit, reviewed_head, diff_sha256, plan_input_sha256, policy_sha256 } } | { status: blocked, request: ReviewRequestEnvelope, reason: string, X: RawLegEvidence | null, S: RawLegEvidence | null, findings: [FindingEvidence], receipt_inputs: { planned_at_commit, reviewed_head, diff_sha256, plan_input_sha256, policy_sha256 } | null }`. plan-manager validates the exact variant and envelope, reconciles accepted/rejected positions without re-running reproduction, and alone writes `## Self-review` or `## Review` + `review_status`. Codex plan-review wrappers use read-only sandboxing; Claude's shipped wrapper cannot enforce plan permission, so it removes Edit/Write tools and writer instructions, retains bounded Bash for reproduction, records pre/post scoped hashes, and STOPs on any mutation. All generated/live plan-review wrappers are return-only; all plan-manager wrappers expose the public review route. No other workflow may launch or reproduce a pair for the same phase/hash.
- **Canonical completion input + sequence:** exact order is: mark steps done → edit `status: in_review` and `in_review_since` → auto-commit that plan-only transition → assert `git status --porcelain -- <plan_path> <affected_paths>` empty → capture `reviewed_head`, canonical plan input, resolved policy/hash, and diff → invoke runner → recheck scoped cleanliness → write verdict/receipt → commit the plan-only review edit. Pathspecs are bytewise UTF-8 sorted by the harness. Diff bytes come from `LC_ALL=C git -c diff.renames=false diff --no-ext-diff --no-textconv --no-color --binary <planned_at_commit>..<reviewed_head> -- <sorted-pathspecs>` and are SHA-256 hashed; nonzero exit or attributes/external filters that alter the no-textconv result STOP. Store compact canonical JSON inside `## Review`: `Completion-review-receipt: {"schema":1,"phase":"completion","request":<ReviewRequestEnvelope>,"planned_at_commit":"<sha>","reviewed_head":"<sha>","diff_sha256":"<hex>","plan_input_sha256":"<hex>","policy":<ResolvedReviewPolicy>,"policy_sha256":"<hex>","X":<persisted-leg>,"S":<persisted-leg>,"outcome":"dual|single|zero-degraded|blocked","reviewed_at":"<ISO>"}`. There is no self-referential `review_commit` field. The later plan-only receipt commit may advance HEAD; ship-time reuse requires every commit after `reviewed_head` to change only this plan and leave canonical non-Review input plus current policy hash unchanged. Any scoped worktree/commit/policy change re-runs completion.

## Out of scope / do-NOT-touch

- Consumer machine env vars, permissions, and a new Docks config file — those belong in DocksDocks/public, not this repo. This plan consumes already-loaded user/runtime-global instructions through the explicit `ReviewPolicyInput`; it does not make shipped skills read arbitrary machine files.
- One user's concrete model/cache preference — the shipped skill carries dated tier defaults plus the logical override input, not a hard-coded personal default.
- How plans are *executed* — only how they are *reviewed/selected*.
- No hard dependency of docks plan-lifecycle on the session-relay plugin; do not edit binaries, manifests, marketplace catalogs, versions, tags, or releases.

## Known gotchas

- Relay `spawn` returns at birth-registration, BEFORE findings exist — never advance the lifecycle on a bare spawn; require `--watch`+bus-reply or use the synchronous CLI leg.
- Two reviewer lists must namespace finding IDs (X#/S#) and preserve per-finding accept/reject reasons, or reconciliation is ambiguous.
- A receipt stored inside the plan can invalidate a naïve whole-file hash. Use the canonical exclusion algorithm exactly; any substantive section change still invalidates it.
- Product consent and host sandbox policy are different signals. `always` is not a sandbox override, and host denial is not permission to try a second export transport.
- A standing consent or zero-review decision is part of the resolved policy hash. A receipt minted under yesterday's policy cannot authorize today's run after any current-user/runtime-global/default tier or provenance change.
- Plugin-shipped Claude agents cannot enforce plan-mode permissions; removing Edit/Write and writer prose is necessary but not sufficient. Their bounded Bash reproduction path must hash scoped inputs before/after and STOP on mutation, while external reviewer legs remain execution-enforced read-only.
- Author company cannot be inferred reliably from the current executor after handoff; persist it when the draft is created and ask once for legacy/unknown plans.
- Ambient model/effort must never be inherited; pins are explicit and dated.
- `timeout 600` is GNU-specific; stock macOS must use the orchestrator's 600-second process/tool timeout and record that evidence.
- A terminal 600-second monotonic deadline is not a transport ETIMEDOUT. Only an execution-layer-typed pre-deadline transport timeout can authorize the one retry.
- Completion review starts only after the `in_review` transition is committed. Capturing HEAD before that commit or reviewing a dirty transition hashes the wrong input.

## Global constraints

- Every plan is independently reviewed by default; availability may degrade the number of legs but never silently erases the receipt.
- Cross-company consent (`always | ask | never`) and zero-review progression (`ask | proceed | block`) are independent decisions.
- Both legs are read-only, findings-only, explicit-model, explicit-effort, fresh-context, and bound to the same immutable commit + canonical input hash.
- Both legs and every receipt echo one request envelope including the exact resolved policy snapshot/hash; any mismatch is invalid evidence.
- plan-manager is the sole lifecycle orchestrator and plan writer; plan-review supplies review mechanics and reproduction, not a competing writer.
- Session-relay is optional; the portable CLI path is normative.
- Host/platform security policy is authoritative and cannot be bypassed by skill text, a user-level receipt, or an alternate transport.
- Skills remain ≤500 lines with current `metadata.updated` + `content_hash`; validators/floors are never weakened.

## STOP conditions

- A reviewer command can write to the repo, inherits ambient permissions/model/effort, or reads a moving worktree instead of the immutable review input.
- A live or generated plan-review wrapper retains Edit/Write or writer instructions, a Codex wrapper is workspace-write, or a bounded-Bash Claude collector changes scoped bytes between its pre/post hashes.
- Cross-company `always` suppresses the zero-review quality decision, or `never` suppresses the same-company leg.
- A receipt reuses after a goal/context/interface/step/acceptance edit, self-invalidates when only its own line/status timestamps change, or lacks exact author/X/S/outcome/time evidence.
- A receipt reuses after any resolved policy/provenance/tier change, or runner/result/leg envelopes disagree on request id, phase, immutable commit/head, input hash, or policy hash.
- Both plan-manager and plan-review can launch/write the same lifecycle review, or lifecycle advances before both legs reach terminal outcomes.
- Completion capture occurs before the plan-only `in_review` commit, with a dirty scoped worktree, or without the post-run cleanliness recheck.
- A platform denial is retried through another transport, an ambiguous failure is labeled `platform_denied`, or a single-subscription user is blocked despite one successful independent leg.
- The contract/template drift, the author-side test becomes a grep/count false positive, or a consumer-side setting/env var is added to this repo.
