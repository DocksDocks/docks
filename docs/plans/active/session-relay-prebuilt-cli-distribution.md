# Deliver Session Relay as a prebuilt CLI

## Goal

Prepare Session Relay as a normal precompiled CLI: the reviewed `0.12.0` source contains only a small launcher, GitHub Actions can build the four supported Rust targets plus `SHA256SUMS`, and a companion reviewed `docks-kit` `0.9.0` source installs an exact host asset whose digest is pinned independently in its repository. This plan also implements and tests prerelease staging, finalization, serialized promotion, smoke receipts, compatibility restoration, and retry machinery. Actual tags, Releases, npm publication, and branch promotion belong to a separate plan created and reviewed only after this source-preparation plan finishes.

## Context & rationale

The current plugin commits four compiled executables and dispatches among them from `plugins/session-relay/bin/relay`. This makes the plugin payload large, obscures provenance during review, and requires an unusual build-artifact commit before every release. The chosen replacement follows the conventional Rust CLI pattern used by projects such as Grok Build: compile per platform in CI, publish immutable checksummed release assets, and let an installer place one host executable on the user's PATH.

The plugin must not download executables from hooks or MCP startup. Those paths are latency-sensitive and operate on untrusted plugin installation state. The plugin launcher therefore performs resolution only and fails with an actionable installation command. `docks-kit` owns downloads, checksum verification, smoke testing, and atomic replacement in the companion repository.

This plan owns the Docks release producer, final launcher-only plugin payload, plugin-consumer contract, resumable release state machine, and immutable handoff to a later release plan. A companion plan in `/home/vagrant/projects/public` owns the `docks-kit` installer and its source-pinned asset schema. Session Relay is the required cross-project implementation transport; the public worker must be dispatched while the current embedded binary still exists. Each repository retains its own plan lifecycle and commit history. The only external mutations allowed here are immutable validation refs for the exact Docks and public source commits; this plan performs no release tag, Release, npm, or production-branch mutation.

## Interfaces & data shapes

### Published release assets

For tag `session-relay--v<VERSION>`, the GitHub Release must contain exactly one executable for every supported target plus one checksum manifest:

```text
session-relay-x86_64-unknown-linux-musl
session-relay-aarch64-unknown-linux-musl
session-relay-x86_64-apple-darwin
session-relay-aarch64-apple-darwin
SHA256SUMS
```

`SHA256SUMS` uses the standard `<64 lowercase hex><two spaces><asset-name>` format and covers all four executables. Asset names are stable and are the contract consumed by `docks-kit`.

### Installed executable

`docks-kit` pins Session Relay `0.12.0` plus one reviewed SHA-256 per supported target in `SoT/toolchain.json` and installs one verified host asset as:

```text
~/.local/bin/session-relay
```

The Rust CLI must support `session-relay --version` and print `session-relay <semver>`. The Cargo package version stays synchronized with the Session Relay plugin manifest version during `scripts/release.mjs`.

The manifest entry is closed data: repository `DocksDocks/docks`, tag `session-relay--v0.12.0`, exact version `0.12.0`, install path `~/.local/bin/session-relay`, and an `assets` map whose four target keys each hold exactly 64 lowercase hex characters. Every ordinary `docks-kit sync` that targets Claude or Codex must ensure the pinned Session Relay CLI before installing, refreshing, or enabling the Session Relay plugin. The ensure path maps the host OS and architecture to one target, downloads the matching executable and `SHA256SUMS`, requires the manifest entry and downloaded bytes both to equal the repository-pinned digest, writes a sibling staged file without replacing the existing command, applies mode `0755`, verifies the exact version from that staged path, and only then atomically renames it into place. An unsupported platform, absent pin, failed download, checksum mismatch, chmod failure, version mismatch, or replacement failure aborts that sync path clearly, removes only its owned stage file, and preserves any existing executable byte-for-byte.

### Plugin launcher resolution

`plugins/session-relay/bin/relay` resolves in this order:

1. Non-empty `SESSION_RELAY_BIN`.
2. `session-relay` found on `PATH`.
3. `$HOME/.local/bin/session-relay`.
4. Exit nonzero with a clear `docks-kit sync` / `docks-kit toolchain ensure session-relay` installation message.

The launcher rejects a resolver target that is itself, preventing recursion. The reviewed and tagged `0.12.0` payload never downloads, compiles, or falls back to an embedded executable. Remote `main` remains on the distinct compatible `0.11.2` payload until `cli-v0.9.0` is live.

### Release orchestration

The binary builder is reusable by tag CI and manual dispatch. On `session-relay--v*`, the validation job must pass before publication. The publisher downloads only artifacts produced by that same workflow run, validates the tag against the manifests and Cargo version, generates `SHA256SUMS`, and creates a public prerelease with the five assets and a staging-only body that explicitly says not to install or advertise `docks-kit sync` yet. The publishing job alone receives `contents: write`; the workflow-level default remains `contents: read`. GitHub's `push` contract binds `GITHUB_SHA` to the pushed tag tip, and a same-repository `./.github/workflows/...` reusable workflow resolves from the same caller commit, so the release producer does not require that commit on the default branch.

The final stable Session Relay GitHub Release install block is exactly `docks-kit sync`, because that command ensures the verified CLI before installing or enabling the plugin for either supported agent runtime. It must not retain the current plugin-only `/plugin install session-relay@docks` instructions. The prerelease body is a separate fixed staging notice with no install command. `scripts/lib/plugins.mjs` owns both descriptors, and the distribution-contract test proves prerelease staging cannot advertise installation while finalization cannot publish plugin-first instructions after embedded binaries are removed.

`scripts/release.mjs --prepare --plugin session-relay 0.12.0` applies every manifest, marketplace, Cargo, and lockfile version change, runs the full CI gate on that exact changed tree, and commits locally without pushing or tagging. A failed post-bump gate prevents the commit. Completion review seals that final launcher-only source. The identity names are fixed: `TAG_COMMIT = SOURCE_COMMIT`, and `PROMOTED_COMMIT = SHIPPED_COMMIT`, where `SHIPPED_COMMIT` is the plan-manager commit that archives this plan with a passed reusable completion receipt. `SOURCE_COMMIT` must be an ancestor of `SHIPPED_COMMIT`, and every path outside this one plan's active/finished lifecycle locations must have the same Git blob or tree identity in both commits. Release-only modes reject an unfinished source-preparation plan. `--publish-reviewed` tags exactly `TAG_COMMIT` and creates only the staged public prerelease. `--promote-reviewed` pushes exactly `PROMOTED_COMMIT`, requires live `cli-v0.9.0`, proves both release identities, creates the durable promotion transaction before its first release mutation, acquires the promotion lock, verifies the expected `origin/main`, downloads and verifies the exact host docks-kit release executable, performs an exact-source pre-push smoke at `TAG_COMMIT`, pushes the reviewed Docks lineage through `PROMOTED_COMMIT`, performs the live smoke, and emits a canonical success receipt. `--resume-promotion` reconciles an interrupted transaction from its permanent remote journal even when no terminal receipt was written. Only `--finalize-reviewed` consumes a successful promotion receipt and changes the exact prerelease to stable plus the final `docks-kit sync` body.

Promotion is one idempotent serialized transaction. Before the lock or any `main` mutation, it creates the permanent append-only remote ref `refs/heads/transactions/session-relay-0.12.0`; every `PromotionJournalEntryV1` commit has exactly one first parent and key `K = (attempt, sequence)`, where `attempt` is a nonnegative integer starting at 0 and `sequence` starts at 0 for each attempt. Keys increase lexicographically with no gap, reuse, or regression. Every entry repeats the immutable source-proof/`TAG_COMMIT`/`PROMOTED_COMMIT`/publication/docks-kit/expected-main identities, prior-entry commit, transaction ref, and promotion-lock identity. Updates use a compare-and-swap lease and never rewrite or delete history. Attempt 0 atomically creates the immutable tombstone `refs/heads/locks/session-relay-0.12.0` at a nonce commit; the lock is never deleted or reused, and every retry reuses that exact transaction and lock. An existing mismatched lock or pre-push drift stops before mutation. The exact-source pre-push smoke runs the released docks-kit executable against a detached reviewed Docks checkout supplied through a release-test-only explicit source argument, proving the launcher-only plugin before `main` changes.

The phase vocabulary is closed to `initialized|locked|prepush_passed|main_pushed|live_passed|restore_pushed|reapply_pushed|terminal_success|terminal_failure|manual_incident`. Attempt 0 has only these legal chains: `initialized -> locked -> prepush_passed -> main_pushed -> live_passed -> terminal_success`; `initialized -> locked -> prepush_passed -> main_pushed -> restore_pushed -> terminal_failure`; or `terminal_failure` from a definitively failed nonterminal phase. When authoritative refs cannot prove one unique safe next phase, the only transition is `manual_incident`. A receipt-authorized retry attempt has only these legal chains: `initialized -> locked -> reapply_pushed -> terminal_success`, where `terminal_success` carries the required post-reapply live-smoke result; `initialized -> locked -> reapply_pushed -> restore_pushed -> terminal_failure`; or `terminal_failure` from a definitively failed nonterminal retry phase. Its initialization records the immediately prior attempt's sealed restored-failure receipt SHA-256 and verified restore commit/head/blob set. `terminal_success`, `terminal_failure`, and `manual_incident` have no outgoing transition within their attempt. No phase may be skipped or repeated, and no retry may execute pre-push publication, tag, Release, npm, or initial-main-push mutations.

Every invocation reconstructs and validates the complete first-parent chain and then reconciles it with authoritative remote refs, GitHub Release identity, exact `origin/main`, closed-set blob IDs, and any restore/reapply commits. A crash never invents a phase: for a nonterminal attempt reconciliation appends the single legal next entry implied by authoritative state or appends `manual_incident` and stops. In attempt 0, pre-push smoke success is journaled before pushing `main`; after a crash, observed `main == PROMOTED_COMMIT` derives `main_pushed`, observed `main == expected-origin-main` derives `prepush_passed`, and any other head is a manual incident. The existing `--resume-promotion` invocation has exactly two disjoint behaviors. For a nonterminal attempt it requires the exact transaction ref plus the same immutable identities and receipt hashes, performs reconciliation, and continues only the unique legal successor. For a terminal tip it performs terminal-receipt recovery only: it requires the exact terminal key and journal commit at the authoritative transaction-ref tip, recomputes the complete chain digest, verifies every immutable identity, promotion-lock identity, authoritative ref and Release identity, exact expected/observed `origin/main`, and the complete before/promoted/restored/reapplied closed-set blob map. It maps `terminal_success` to `success`, `manual_incident` to `manual_incident`, a `terminal_failure` with a verified authoritative restore commit and restored closed-set map to `restored_failure`, and any other valid terminal failure to non-retryable `failure`. Every receipt field, including its creation time, is copied or deterministically derived from the terminal journal payload and validated authoritative state, so uninterrupted emission and recovery produce byte-identical RFC 8785 JCS. Terminal recovery never appends a journal entry, changes phase, pushes, restores, reapplies, runs a smoke, or repeats any mutation. If the filesystem path named by `--receipt-out` is absent, recovery atomically creates the mode-`0600` receipt through an exclusive mode-`0600` temporary sibling; if that path already contains the exact expected bytes at mode `0600`, recovery succeeds idempotently; any other existing bytes, type, mode, hash, terminal outcome, or identity fail closed without mutation.

The compatibility restore/reapply set is closed to the entire tracked `plugins/session-relay/**` tree plus the whole `.claude-plugin/marketplace.json` blob. `PromotionReceiptV1` records before, promoted, restored, and reapplied Git blob IDs for every member. Shared `.github/**` and `scripts/**` changes deliberately remain because their contract tests prove compatibility with the restored `0.11.2` payload and unchanged docks/effect-kit release semantics. A definitive post-push failure first creates a normal fast-forward restore commit atop the locked `main`, restoring every closed-set member from `expected-origin-main` after proving each still equals its promoted blob, and the restore must pass manifest/catalog parity and exact full CI before push. Restore failure appends `terminal_failure` with the attempted restore identity and no reapply; an unverifiable restore, unexpected main movement, or concurrent closed-set change appends `manual_incident`, keeps the prerelease staged, and stops finalization.

Receipt-based retry exists only as `--retry-failed <receipt-path> --retry-failed-sha256 <sha256>` against the latest journal tip. It accepts only a canonical retryable `restored_failure` receipt—whether emitted uninterrupted or byte-identically recovered by `--resume-promotion`—whose `attempt` equals the terminal attempt, whose terminal journal commit/key and chain digest equal the permanent transaction tip, whose prior-receipt binding and immutable identities match, whose exact restore commit is still remote `main`, and whose closed-set blobs remain unchanged. Of the recoverable outcomes, only that retryable `restored_failure` authorizes one next retry attempt; recovered `success`, `failure`, `manual_incident`, and non-retryable restored failures never do. It rejects non-retryable or older receipts, a nonterminal/forked/regressed journal, mismatched attempts or keys, missing/extra commits, a restore-head mismatch, and every intervening closed-set change. Acceptance increments `attempt` exactly once, appends the next `initialized` entry with the recovered receipt SHA-256 as `prior_attempt_receipt_sha256` under compare-and-swap, and then follows the closed retry chain. Reusing that receipt after the next attempt exists cannot append another `initialized` entry or repeat reapply; it resolves only the already-bound attempt or fails closed. A definitive failure after that reapply is terminal and non-retryable for `0.12.0`; only a separately reviewed `0.12.1` transaction may supersede it. Force-push, journal or transaction-ref deletion, lock replacement, and history rewriting are forbidden.

Publication is independently resumable. `--publish-reviewed` reconciles these states against the exact reviewed commit and tag:

| State | Required behavior |
|---|---|
| Tag absent | Create and push the immutable tag, then wait for its bound tag-push publication run. |
| Matching tag, no bound run, no Release | Dispatch `publish-existing-tag` against the exact tag/ref and commit; any mismatch stops. |
| Matching tag and bound run, no Release | Resume that run or create the prerelease from artifacts produced by that exact run. |
| Matching prerelease, partial assets | Verify every existing asset's name and digest, upload only missing assets from the bound run, then verify the closed five-asset set and staging body. |
| Matching complete prerelease | Return success without replacing assets. |
| Matching stable Release | `--publish-reviewed` returns a typed premature-stable conflict. Stable idempotence belongs only to `--finalize-reviewed` with the bound successful promotion and final-publication receipts. |
| Tag, run, Release ID, target commit, asset, or digest conflict | Stop with a typed conflict result; never clobber, retag, or substitute another run. |

`--finalize-reviewed` treats a matching complete prerelease plus a successful promotion receipt as actionable and an already stable matching Release/body as success; absent/partial/conflicting state or a failure receipt stops. Every transition binds repository ID, tag object/commit, workflow file/SHA/run ID/attempt, Release database ID, prerelease state/body hash, and asset database IDs/digests. Contract tests simulate interruption at every row and prove convergence or fail-closed conflict behavior.

### Release evidence schemas

Release evidence uses RFC 8785 JSON Canonicalization Scheme UTF-8 bytes with SHA-256 over the exact canonical bytes. Every schema is closed, versioned with `schema: 1` plus an exact `type`, rejects unknown/missing fields, and is written only to an explicit `--receipt-out` path. Writers create a mode-`0600` temporary sibling with exclusive create, write and fsync the canonical bytes, atomically rename, fsync the parent directory, then print the lowercase receipt SHA-256. Validators require the caller-provided path plus SHA-256 and never search ambient directories.

- `TddRedReceiptV1`: repository ID, repository commit before production edits, test paths/blob IDs, exact command/working-directory identity, required nonzero exit, stdout/stderr SHA-256, capture timestamp, and producer version. Docks and public each emit one canonical receipt immediately after freezing their tests; the exact bytes and hashes are embedded under Notes, later validation requires the recorded pre-production commit to be an ancestor of the implementation commit, and every recorded test path must have exactly the recorded blob ID there.
The fixed Notes encoding is part of the interface: each `... receipt JCS bytes: ` prefix is followed on the same line by exactly one RFC 8785 JSON object, and the embedded receipt byte sequence is the UTF-8 bytes after that prefix through but excluding the line-feed delimiter. Its adjacent `... receipt SHA-256: ` field is exactly the lowercase SHA-256 of that byte sequence. `pending`, duplicate fields, carriage returns, noncanonical JSON, a repository-ID mismatch, or any extra byte are invalid.
- `ProducerPreflightReceiptV1`: repository ID, validation-only branch/ref commit, workflow file/SHA/run database ID/attempt/event/input object/conclusion, four artifact database IDs/names/digests/file identities, four native-attestation database IDs with runner OS/architecture and exact version outputs, checksum artifact/database ID/manifest digest, verifier version, and creation timestamp.
- `SourceCiReceiptV1`: repository ID, `SOURCE_COMMIT`, workflow file/SHA/run database ID/attempt/event/conclusion, Node/pnpm/Claude/Rust/musl bootstrap identities, frozen-lockfile identity, exact command `node scripts/ci.mjs`, job/log SHA-256, and start/end timestamps.
- `SourcePreparationCandidateV1`: repository ID, plan path/blob SHA-256 before evidence embedding, execution-base commit, `SOURCE_COMMIT`, Docks red-receipt SHA-256 and test blob identities, companion repository ID/immutable validation ref/commit/plan path/input SHA-256/execution-base/review-receipt SHA-256/red-receipt SHA-256/status/blocked reason, preflight-receipt and source-CI-receipt SHA-256 plus every producer identity, internally rerun A1–A6 stdout/stderr/exit hashes, and creation timestamp. `--check-prepared` consumes the two explicit red receipts plus preflight and source-CI receipts, freshly fetches the public validation ref and requires its commit, reruns A1–A6 itself, verifies both frozen test identities and the complete release-owned tree at `SOURCE_COMMIT`, and emits the candidate before completion review.
- `SourcePreparationProofV1`: every candidate field plus candidate SHA-256, final plan path/blob SHA-256, exact reviewed plan-only evidence commit, `TAG_COMMIT` equal to unchanged `SOURCE_COMMIT`, `PROMOTED_COMMIT` equal to `SHIPPED_COMMIT`, their required ancestry and non-plan tree-equivalence proof, embedded reusable completion-review receipt SHA-256 and `review_status: passed`, and binding timestamp. The later release plan begins with `--bind-completion --finished-plan <path> --embedded-candidate-sha256 <sha256> --receipt-out <path>`; it emits the proof only when the finished plan and receipt bind the evidence commit, the complete release-owned tree still equals `SOURCE_COMMIT`, and the promoted descendant differs only at this plan's lifecycle path.
- `SessionRelayPublicationReceiptV1`: source-proof SHA-256, repository/tag/`TAG_COMMIT` (exactly `SOURCE_COMMIT`), workflow file/SHA/run ID/attempt, Release database ID and `prerelease|stable` state, body SHA-256, five closed asset records with database ID/name/size/digest, transition performed, and timestamp. Publish/finalize modes consume the prior receipt when resuming and emit the reconciled state.
- `PromotionJournalEntryV1`: permanent transaction ref; exact `attempt`, `sequence`, and prior-entry commit (null only at `(0,0)`); immutable source-proof/`TAG_COMMIT`/`PROMOTED_COMMIT`/publication/docks-kit/expected-main identities; immutable promotion-lock ref/nonce; closed phase; prior-attempt receipt SHA-256 (null only on attempt 0); observed remote main and Release state; relevant push/restore/reapply commits and closed-set blob IDs; smoke descriptor/result hashes when available; and timestamp. The current entry commit is derived from the Git object containing the canonical payload and is never included in that payload. Validators reconstruct the full first-parent chain, require gap-free lexicographic keys and one of the exact attempt-0 or retry chains, and reject any outgoing transition from a terminal entry.
- `PromotionReceiptV1`: source-proof and publication-receipt SHA-256, exact `TAG_COMMIT` and `PROMOTED_COMMIT` plus ancestry/non-plan equivalence proof, transaction ref, exact `attempt`, terminal key and terminal journal commit, journal-chain digest, prior-attempt receipt SHA-256 (null only on attempt 0), promotion-lock ref/nonce, expected and observed `origin/main`, Docks pre-push/pushed/restore/reapply commits, closed compatibility-set paths and before/promoted/restored/reapplied Git blob IDs, public reviewed/tag commits and Release ID, docks-kit and Session Relay asset identities, exact-source/live isolation-root descriptors and sync argv, stdout/stderr/ordering-log hashes, installed binary and launcher identities/version outputs, post-restore parity/full-CI evidence, outcome `success|restored_failure|failure|manual_incident`, retryability, and terminal-entry-derived creation time. The terminal journal entry carries the complete receipt projection needed to reproduce those fields; only terminal commit identity and chain digest are derived from the validated chain. Recovery status is not encoded in the receipt. The receipt is atomically written after the terminal journal entry, and `--resume-promotion` reconstructs these same unique canonical bytes when the write was interrupted.
